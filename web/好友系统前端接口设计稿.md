# 好友系统前端接口设计稿

## 1. 文档目标

这份文档用于给现有 `web` 前端新增好友系统提供稳定的接口契约，目标是：

- 明确前端需要哪些 HTTP API
- 明确每个 API 的请求字段、响应字段、状态语义
- 统一字段命名风格，避免前后端出现 `camelCase / snake_case` 混用
- 预留好友邀请对战的实时通知协议
- 让后端可以据此直接开始实现接口
- 让前端后续可以直接按本稿新增 `API_PATHS` 与 `friend store`

本稿基于当前项目已有接口风格整理：

- 认证方式沿用 JWT：`Authorization: Bearer <token>`
- HTTP 参数风格沿用现有项目：优先 `@RequestParam` 兼容的表单/query 风格
- 路径风格沿用现有接口：以 `/api/.../` 结尾
- 返回结构沿用现有项目：不强制统一 `code/data/msg` 包装，但对好友系统建议统一结构，降低前端复杂度

## 2. 设计原则

### 2.1 字段命名原则

建议好友系统所有 HTTP 接口统一使用 `snake_case` 字段命名。

原因：

- 当前项目已有大量表单字段使用 snake_case，例如：
- `new_username`
- `old_password`
- `confirmed_password`
- `bot_id`
- 这样可以减少前端 `apiRequest` 封装层额外映射成本
- 后端如果使用 Spring `@RequestParam`，snake_case 直传最稳

### 2.2 返回结构原则

当前项目历史接口风格不完全统一，但好友系统建议统一成如下结构：

```json
{
  "error_message": "success",
  "data": {}
}
```

或列表型：

```json
{
  "error_message": "success",
  "friends": [],
  "friends_count": 0
}
```

即：

- 保留当前项目通用字段 `error_message`
- 避免再引入新的 `code` / `message` / `success` 多套体系
- 列表接口直接返回语义化列表字段

### 2.3 关系模型原则

好友系统建议拆成 3 套核心对象：

- 好友关系 `Friend`
- 好友申请 `FriendRequest`
- 好友对战邀请 `FriendInvite`

不要把“好友”和“申请”混在一个对象里，否则前端状态会非常别扭。

## 3. 核心业务对象

## 3.1 Friend

表示“已经建立好友关系”的用户对象。

### 推荐结构

```json
{
  "friend_id": 12,
  "friend_username": "alice_01",
  "friend_photo": "http://127.0.0.1:3000/avatars/a.png",
  "friend_rating": 1678,
  "online_status": "online",
  "last_active_at": "2026-04-15 20:10:23",
  "last_match_at": "2026-04-14 19:05:11",
  "match_count": 8,
  "is_favorite": false,
  "created_at": "2026-04-01 14:22:31"
}
```

### 字段说明

- `friend_id`：好友用户 ID
- `friend_username`：好友用户名
- `friend_photo`：好友头像 URL
- `friend_rating`：好友当前积分
- `online_status`：在线状态
- `last_active_at`：最后活跃时间
- `last_match_at`：最近一次和我对战时间，可为空
- `match_count`：历史共同对战次数
- `is_favorite`：是否被我标记为常用好友，可选
- `created_at`：成为好友的时间

### `online_status` 建议取值

- `online`
- `in_game`
- `offline`
- `busy`

如果第一版后端做不到全部细分，至少保证：

- `online`
- `offline`

## 3.2 FriendRequest

表示“好友申请”。

### 推荐结构

```json
{
  "request_id": 103,
  "sender_id": 5,
  "sender_username": "bob_88",
  "sender_photo": "http://127.0.0.1:3000/avatars/b.png",
  "sender_rating": 1590,
  "receiver_id": 12,
  "receiver_username": "alice_01",
  "receiver_photo": "http://127.0.0.1:3000/avatars/a.png",
  "receiver_rating": 1678,
  "message": "来打一局？",
  "status": "pending",
  "created_at": "2026-04-15 18:12:10",
  "handled_at": null
}
```

### 字段说明

- `request_id`：申请记录 ID
- `sender_*`：申请发起人信息
- `receiver_*`：申请接收人信息
- `message`：申请附言，可为空
- `status`：申请状态
- `created_at`：申请发起时间
- `handled_at`：处理时间，未处理时为空

### `status` 建议取值

- `pending`
- `accepted`
- `ignored`
- `cancelled`
- `expired`

前端常用的是：

- 收到的申请里主要关心 `pending`
- 发出的申请里主要关心 `pending` / `accepted` / `cancelled`

## 3.3 FriendInvite

表示“好友对战邀请”。

### 推荐结构

```json
{
  "invite_id": 55,
  "sender_id": 12,
  "sender_username": "alice_01",
  "sender_photo": "http://127.0.0.1:3000/avatars/a.png",
  "receiver_id": 5,
  "receiver_username": "bob_88",
  "game_mode": "pk",
  "sender_bot_id": -1,
  "sender_bot_title": "manual",
  "status": "pending",
  "created_at": "2026-04-15 20:20:10",
  "expired_at": "2026-04-15 20:20:40"
}
```

### 字段说明

- `invite_id`：邀请 ID
- `sender_id`：邀请方用户 ID
- `receiver_id`：被邀请方用户 ID
- `game_mode`：当前先固定 `pk`
- `sender_bot_id`：邀请方本局出战 Bot，`-1` 表示手动
- `sender_bot_title`：前端展示用，可选
- `status`：邀请状态
- `created_at`：创建时间
- `expired_at`：过期时间

### `status` 建议取值

- `pending`
- `accepted`
- `rejected`
- `cancelled`
- `expired`

## 4. 接口清单总览

建议新增以下 HTTP 接口：

- `GET /api/friends/list/`
- `GET /api/friends/stats/`
- `GET /api/friends/search/`
- `POST /api/friends/request/send/`
- `GET /api/friends/request/received/`
- `GET /api/friends/request/sent/`
- `POST /api/friends/request/accept/`
- `POST /api/friends/request/ignore/`
- `POST /api/friends/request/cancel/`
- `POST /api/friends/remove/`
- `POST /api/friends/favorite/toggle/` 可选
- `POST /api/friends/invite/send/`
- `POST /api/friends/invite/respond/`
- `GET /api/friends/invite/pending/` 可选，用于页面刷新恢复状态

## 5. 详细接口设计

## 5.1 获取好友列表

### 接口

- 方法：`GET`
- 路径：`/api/friends/list/`
- 鉴权：需要 JWT

### 请求参数

- `page`：页码，可选；默认 1
- `page_size`：分页大小，可选；默认 20
- `keyword`：按用户名模糊过滤，可选
- `status`：按在线状态筛选，可选
- `favorite_only`：是否只看收藏好友，可选，`true/false`
- `sort_by`：排序方式，可选

### `sort_by` 建议取值

- `created_at_desc`
- `last_active_desc`
- `last_match_desc`
- `rating_desc`

### 返回示例

```json
{
  "error_message": "success",
  "friends": [
    {
      "friend_id": 12,
      "friend_username": "alice_01",
      "friend_photo": "http://127.0.0.1:3000/avatars/a.png",
      "friend_rating": 1678,
      "online_status": "online",
      "last_active_at": "2026-04-15 20:10:23",
      "last_match_at": "2026-04-14 19:05:11",
      "match_count": 8,
      "is_favorite": false,
      "created_at": "2026-04-01 14:22:31"
    }
  ],
  "friends_count": 31
}
```

### 前端用途

- 左侧好友列表
- 在线好友筛选
- 最近对战好友筛选
- 搜索框本地结果承接

## 5.2 获取好友统计信息

### 接口

- 方法：`GET`
- 路径：`/api/friends/stats/`

### 请求参数

- 无

### 返回示例

```json
{
  "error_message": "success",

    "friends_count": 31,
```

### 前端用途

- 导航栏未读申请徽标

## 5.3 搜索用户

### 接口

- 方法：`GET`
- 路径：`/api/friends/search/`
- 鉴权：需要 JWT

### 请求参数

- `keyword`：用户名或用户 ID，必填
- `page`：可选
- `page_size`：可选，默认 10

### 返回示例

```json
{
  "error_message": "success",
  "users": [
    {
      "user_id": 18,
      "username": "charlie_9",
      "photo": "http://127.0.0.1:3000/avatars/c.png",
      "rating": 1542,
      "online_status": "offline",
      "relation_status": "none",
      "request_id": null
    }
  ],
  "users_count": 1
}
```

### `relation_status` 建议取值

- `self`
- `friend`
- `none`
- `sent_pending`
- `received_pending`
- `blocked` 可选

### 前端用途

- 发现页搜索结果
- 搜索结果里直接决定按钮状态

### 关键说明

搜索结果里最好直接带上 `relation_status`，不要让前端自己再做多次接口拼装。

## 5.4 发送好友申请

### 接口

- 方法：`POST`
- 路径：`/api/friends/request/send/`
- 鉴权：需要 JWT

### 请求参数

- `receiver_id`：目标用户 ID，必填
- `message`：申请附言，可选，最多 50-100 字

### 请求示例

```text
receiver_id=18&message=来打一局？
```

### 返回示例

```json
{
  "error_message": "success",
  "request": {
    "request_id": 103,
    "status": "pending"
  }
}
```

### 常见失败返回建议

```json
{ "error_message": "cannot add yourself" }
```

```json
{ "error_message": "already friends" }
```

```json
{ "error_message": "request already sent" }
```

### 前端用途

- 搜索结果点击“添加好友”
- 好友详情页中的“加好友”动作

## 6. WebSocket 设计建议

好友系统涉及两类实时事件：

- 好友申请变化
- 好友对战邀请

### 6.1 推荐方案

优先建议复用现有主 WebSocket：

```text
/ws://host:3000/websocket/{token}
```

原因：

- 当前项目已经有登录后的主 socket
- 少建一条 WebSocket 可降低前端连接复杂度
- 好友邀请本质上和 PK 流程强相关，放在主 socket 更自然

### 6.2 建议新增事件

#### 事件：收到新的好友申请

服务端推送：

```json
{
  "event": "friend-request-received",
  "request": {
    "request_id": 103,
    "sender_id": 5,
    "sender_username": "bob_88",
    "sender_photo": "http://127.0.0.1:3000/avatars/b.png",
    "sender_rating": 1590,
    "message": "来打一局？",
    "status": "pending",
    "created_at": "2026-04-15 18:12:10"
  }
}
```

#### 事件：好友申请被处理

服务端推送：

```json
{
  "event": "friend-request-handled",
  "request_id": 103,
  "status": "accepted"
}
```

#### 事件：收到对战邀请

服务端推送：

```json
{
  "event": "friend-invite-received",
  "invite": {
    "invite_id": 55,
    "sender_id": 12,
    "sender_username": "alice_01",
    "sender_photo": "http://127.0.0.1:3000/avatars/a.png",
    "sender_bot_id": -1,
    "sender_bot_title": "manual",
    "status": "pending",
    "created_at": "2026-04-15 20:20:10",
    "expired_at": "2026-04-15 20:20:40"
  }
}
```

#### 事件：对战邀请状态变化

服务端推送：

```json
{
  "event": "friend-invite-updated",
  "invite_id": 55,
  "status": "accepted"
}
```

### 6.3 前端处理建议

在现有 `PkIndexView.vue` WebSocket 消息路由基础上，新增对这些事件的分支处理即可。

建议不要单独新建好友 socket，除非后端明确希望把主对战与社交事件彻底分离。

## 7. 前端 Store 建议映射

建议新建 `friend.js` store 时，对应维护以下状态：

```js
{
  friends: [],
  friendsCount: 0,
  stats: {
    friends_count: 0,
    online_count: 0,
    pending_received_count: 0,
    pending_sent_count: 0,
    today_new_count: 0,
  },
  receivedRequests: [],
  sentRequests: [],
  searchResults: [],
  selectedFriendId: null,
  pendingInvites: [],
  loading: {
    friends: false,
    requests: false,
    search: false,
    invite: false,
  }
}
```

### 建议 action 命名

- `fetchFriends()`
- `fetchFriendStats()`
- `searchUsers(keyword)`
- `sendFriendRequest(receiverId, message)`
- `fetchReceivedRequests()`
- `fetchSentRequests()`
- `acceptFriendRequest(requestId)`
- `ignoreFriendRequest(requestId)`
- `cancelFriendRequest(requestId)`

## 9. 错误语义建议

为了减少前端分支判断复杂度，建议错误信息尽量稳定。

### 推荐常用错误文案

- `success`
- `invalid params`
- `friend not found`
- `request not found`
- `invite not found`
- `already friends`
- `request already sent`
- `cannot add yourself`
- `friend is offline`
- `friend is in game`
- `invite already exists`
- `permission denied`

前端可以直接按 `error_message !== "success"` 做统一失败处理。

## 10. 分页与性能建议

### 10.1 好友列表

- 默认分页，每页 20 条
- 如果好友数不大，也可以第一版直接全量返回
- 但接口保留 `page/page_size`，避免后期破坏兼容性

### 10.2 搜索接口

- 建议支持分页
- 前端做 300ms debounce
- 后端对空关键词直接返回空列表，不要全表扫

### 10.3 申请列表

- 允许分页，但第一页优先返回 `pending` 项

## 11. 前后端对齐建议

### 11.1 后端最好直接返回前端展示所需字段

例如搜索用户时，不要只返回裸用户表字段，还要附带：

- `relation_status`
- `request_id` 可选
- `online_status`

否则前端为了渲染一个按钮，要多打很多请求。

### 11.2 邀请逻辑建议服务端最终仍走现有开局链路

即：

- 好友邀请被接受后
- 后端内部还是调用现有的匹配/开局机制，或直接走 `/pk/start/game/`

前端只关心：

- 我是否已接受邀请
- 是否收到了进入对战的 socket 事件

### 11.3 申请与好友关系应有唯一性约束

建议后端保证：

- 同一对用户不能重复存在有效好友关系
- 同一对用户同一时间只能存在一条 `pending` 申请
- 同一对用户同一时间只能存在一条 `pending` 邀请

这样前端按钮状态才能稳定。

## 12. 最小可用版本建议

如果后端想分阶段实现，建议第一阶段先做：

1. `GET /api/friends/list/`
2. `GET /api/friends/stats/`
3. `GET /api/friends/search/`
4. `POST /api/friends/request/send/`
5. `GET /api/friends/request/received/`
6. `GET /api/friends/request/sent/`
7. `POST /api/friends/request/accept/`
8. `POST /api/friends/request/ignore/`
9. `POST /api/friends/request/cancel/`
10. `POST /api/friends/remove/`

第二阶段再补：

- 好友邀请对战
- 在线状态细分
- 常用好友/收藏好友
- WebSocket 实时通知

## 13. 结论

这套接口设计稿的核心目标不是“把好友系统做复杂”，而是把前端最需要的三件事先定清：

1. 数据对象拆分清楚：好友、申请、邀请分离
2. 列表和搜索结果直接返回可渲染字段
3. 邀请对战复用现有主 WebSocket 与开局链路

这样后续前端可以直接：

- 在 `env.js` 中补 `API_PATHS`
- 新建 `friend.js` store
- 接好友主页、申请页、搜索页、邀请弹层
- 在 `PkIndexView.vue` 中补好友邀请相关 socket 事件分发

本稿可以作为后端实现好友系统 API 的第一版契约文档。
