# 新增好友系统前端方案与 UI/UX 交互方案

## 1. 目标与设计基准

本方案基于当前项目已有前端风格进行扩展，不另起一套视觉体系。

当前前端已形成的视觉语言：

- 轻浅色背景 + 冷蓝主色 + 少量暖黄点缀
- 全局玻璃卡片感：半透明白底、轻描边、低强度阴影、模糊背景
- 标题字体使用 `Space Grotesk`，正文以 `IBM Plex Sans` 为主
- 强交互按钮多为圆角胶囊按钮或圆角 12px/16px 按钮
- 页面主要由 `ContentField` 作为通用承载卡片
- 列表页已经有成熟的 table + skeleton + empty + pagination 样式
- 浮层、确认弹窗已经有 `AppDialog` 作为统一能力
- 动效节奏整体偏轻，过渡速度短，强调状态反馈而非炫技

因此，好友系统必须满足：

- 视觉上是“现有页面的自然延伸”
- 信息结构清晰，不抢 PK 主玩法的主视觉中心
- 支持桌面与移动端
- 保持一套统一的状态反馈逻辑：空态、加载、提交中、成功、失败、撤销

## 2. 设计方向

### 2.1 产品定位

好友系统不是独立社交模块，而是为现有 PK 游戏补上“熟人对战”和“关系留存”能力。

所以它的定位不是做复杂社区，而是做一个：

- 足够轻量
- 足够直接
- 足够顺手
- 不打断主对战路径

的关系系统。

### 2.2 体验原则

围绕三个原则设计：

1. 先让用户“找到人”
2. 再让用户“建立关系”
3. 最后让用户“从关系进入对战”

好友系统不应该把用户带到一个完全新世界，而应该成为：

- 导航里的一个稳定入口
- 对战页里的一个快捷入口
- 用户资料卡里的一个关系入口

## 3. 信息架构

推荐新增一个一级页面：`好友`

### 3.1 导航层级建议

当前导航：

- `PK`
- `对局记录`
- `排行榜`

建议新增为：

- `PK`
- `好友`
- `对局记录`
- `排行榜`

原因：

- 好友是高频操作，应该成为一级入口，而不是藏在用户下拉菜单里
- 当前一级导航只有 3 项，增加到 4 项仍然清晰
- 它和 PK 的关联比和“我的 Bot”更强，更适合放在主导航区

### 3.2 页面结构建议

新增主页面路由：

- `/friends/`
- 路由名：`friends_index`

页面内部采用三分区结构：

1. 顶部关系总览区
2. 主内容双栏区
3. 辅助弹层系统

其中主内容双栏建议如下：

- 左栏：好友列表 / 分组 / 在线状态 / 快捷操作
- 右栏：搜索添加好友 / 申请列表 / 关系详情 / 空态引导

桌面端：

- 左栏固定占 4 栅格
- 右栏占 8 栅格

移动端：

- 上方改为分段切换 Tabs
- Tabs：`好友` / `申请` / `发现`

## 4. 页面方案

## 4.1 好友主页 `FriendsIndexView`

### 页面目标

让用户一进页面就知道：

- 我现在有多少好友
- 有多少待处理申请
- 哪些好友在线
- 我可以从哪里快速加好友

### 页面布局

#### A. 顶部总览条

样式延续当前 `ContentField + panel-header` 语言，但更轻更横向。

建议包含 4 张小型统计卡：

- `好友数`
- `在线好友`
- `待处理申请`
- `今日新增`

视觉样式：

- 卡片背景仍为 `var(--kob-panel)`
- 数字使用 `Space Grotesk` 粗字重
- 在线数字可用绿色小点配合显示
- 待处理申请可使用蓝色徽标强调，不使用刺眼红色常驻提示

#### B. 左侧好友栏

包含：

- 搜索框
- 好友分组切换
- 好友列表

分组建议：

- `全部好友`
- `在线`
- `最近对战`
- `已收藏`

如果当前后端暂时不支持分组，也可以先作为前端筛选维度。

#### C. 右侧内容区

右侧区根据当前选中的 tab / 子状态切换：

- 默认显示“好友详情 / 快捷操作面板”
- 切到 `申请` 时显示待处理申请列表
- 切到 `发现` 时显示搜索添加与推荐用户

## 4.2 好友详情面板

### 目标

当用户点击左侧某位好友时，右侧面板替换为该好友详情。

### 内容建议

上半部分：

- 大头像
- 用户名
- Rating
- 在线状态
- 最近对战时间

中部快捷操作区：

- `邀请对战`
- `发消息`
- `设为常用`
- `删除好友`

下半部分可选扩展：

- 最近战绩摘要
- 共同对战次数
- 上次一起对局时间

### 设计处理

这块不应该做成厚重资料页，而应做成轻量作战面板，强调“下一步动作”。

## 4.3 待处理申请页块

### 内容分两类

- 我收到的好友申请
- 我发出的好友申请

建议用 segmented control 或 tab pills 切换：

- `收到的申请`
- `发出的申请`

### 每个申请卡片包含

- 头像
- 用户名
- Rating
- 申请附言
- 发送时间
- 操作按钮

收到的申请操作：

- `接受`
- `忽略`

发出的申请操作：

- `撤回`

### 视觉处理

收到申请卡片建议比普通好友卡更强调“待处理感”：

- 边框更亮一点
- 顶部可带小标签 `待处理`
- 操作按钮更明确

但不建议通篇大红色，会破坏当前页面轻快氛围。

## 4.4 搜索添加好友区

### 搜索框位置

建议放在右栏 `发现` 面板顶部，而不是全局弹窗。

### 搜索模式

支持至少两类：

- 按用户名搜索
- 按 ID 精确搜索

### 搜索结果项展示

每项展示：

- 头像
- 用户名
- Rating
- 关系状态
- 操作按钮

状态种类：

- `不是好友` -> `添加好友`
- `申请已发送` -> `已发送`
- `对方已向你申请` -> `查看申请`
- `已经是好友` -> `已是好友`
- `自己` -> 禁止操作

### 搜索体验建议

- 输入 300ms debounce
- 空输入不请求
- 少于 2 个字符只提示“继续输入”
- 搜索结果列表高度固定，避免页面大幅跳动

## 4.5 PK 页内的好友快捷入口

这是最重要的联动设计。

### 现有匹配页结构

当前 `MatchGround` 是一个三列结构：

- 自己头像
- Bot 选择
- 对手头像

底部是开始匹配按钮。

### 推荐新增一个轻量好友快捷模块

放置位置：

- `MatchGround` 中部 Bot 选择下方
- 或主按钮上方一行

文案结构：

- 左侧文本：`想和好友直接开打？`
- 右侧按钮：`邀请好友`

点击后打开“好友邀请抽屉 / 弹层”。

### 为什么不直接塞进主页面常驻

- PK 页的主目标仍是开始匹配
- 好友邀请是次级路径，不应该抢主 CTA
- 用一条轻提示 + 弹层，比直接加大块面板更干净

## 6. 视觉与样式方案

## 6.1 核心视觉延续

必须复用现有全局 token：

- `--kob-panel`
- `--kob-panel-border`
- `--kob-text`
- `--kob-muted`
- `--kob-accent`
- `--kob-accent-strong`
- `--kob-warn`

新增建议 token：

- `--kob-success: #27ae60`
- `--kob-success-soft: rgba(39, 174, 96, 0.12)`
- `--kob-danger-soft: rgba(231, 76, 60, 0.12)`
- `--kob-friend-active: rgba(61, 174, 255, 0.14)`
- `--kob-online-ring: rgba(39, 174, 96, 0.55)`

### 6.2 好友卡片视觉

好友列表项建议采用：

- 白玻璃浅卡片
- 左侧头像 + 在线状态小点
- 中间用户名与副信息
- 右侧一个次级操作按钮或箭头

选中态处理：

- 背景由透明变为浅蓝高亮
- 外边框加强
- 右侧出现 2px 的蓝色激活条

不要使用深色整块选中，否则会与当前整体轻盈氛围冲突。

### 6.3 状态标签设计

标签风格统一成小圆角胶囊：

- `在线`：绿色浅底 + 绿色文字
- `待处理`：浅蓝底 + 蓝字
- `已发送`：浅黄底 + 深黄字
- `最近一起玩过`：浅灰蓝底 + 中性色字

### 6.4 字体层级

建议保留当前组合：

- 页面标题：`Space Grotesk`
- 模块标题：`Space Grotesk`
- 正文、元信息、操作说明：`IBM Plex Sans`

层级建议：

- 页标题：28px / 700
- 模块标题：18px / 700
- 列表昵称：15px / 600
- 辅助信息：12px / 500
- 标签文字：11px / 700

## 7. 关键 UI 交互逻辑

## 7.1 添加好友主流程

### 流程

1. 用户进入 `好友` 页
2. 切到 `发现`
3. 输入用户名搜索
4. 结果列表出现
5. 点击 `添加好友`
6. 出现轻量附言输入层
7. 提交后按钮变为 `已发送`
8. 顶部 toast 提示 `好友申请已发送`

### 交互细节

- 搜索结果出现使用 180ms 淡入 + 轻微上移
- 提交按钮进入 loading，防止重复点击
- 成功后当前卡片原位状态切换，不整页刷新
- 若失败，错误信息贴近按钮下方或以 toast 呈现

### 设计理由

这里要强调“连续性”，不能点一次后整个搜索面板重刷，会很割裂。

## 7.2 接收好友申请流程

### 流程

1. 用户进入 `申请`
2. 收到的申请列表顶部默认展示未处理项
3. 点击 `接受`
4. 卡片进入 loading
5. 成功后卡片从申请列表退出
6. 自动插入左侧好友列表顶部
7. 顶部统计条 `好友数 +1`
8. 顶部统计条 `待处理申请 -1`
9. 右下轻提示：`已添加为好友，现在可以邀请对战`

### 动效建议

- 接受成功时，申请卡片渐隐收起
- 左侧好友列表顶部插入新好友时做 220ms 高度展开 + fade-in
- 这是一个非常重要的“状态闭环”反馈

## 7.3 删除好友流程

### 流程

1. 在好友详情面板点击 `删除好友`
2. 使用现有 `AppDialog` 做二次确认
3. 确认后进入 loading
4. 删除成功后：
- 当前详情面板切回空态
- 左侧列表移除该用户
- 顶部统计同步更新
- toast 提示 `已移除好友`

### UX 要求

- 删除操作必须二次确认
- 删除成功后要提供短时可恢复入口更好，如果后端暂不支持撤销，则只做提示

## 7.4 邀请好友对战流程

这是整个好友系统最有价值的动作。

### 推荐流程

1. 在好友详情点击 `邀请对战`
2. 打开 `FriendInviteDialog`
3. 弹层中选择出战方式：
- `手动操作`
- `选择我的 Bot`
4. 确认发送邀请
5. 被邀请方在全局顶部或页内收到邀请条
6. 被邀请方可：
- `接受`
- `拒绝`
- `稍后`
7. 接受后直接跳入 PK 房间/对战初始化流程

### UI 建议

邀请弹层不要做成表单密集型，而应像战术卡片：

- 标题：`邀请好友对战`
- 中部：好友头像、名称、当前状态
- 下部：Bot 选择器 + 操作按钮

### 动效建议

- 邀请弹层从触发按钮位置附近放大浮现，300ms
- 接收邀请通知使用顶部浮动条，从导航栏下方滑入
- 接受后整个条目压缩消失，给人“动作已处理”的明确反馈

## 7.5 在线状态交互逻辑

### 状态层级

建议至少区分：

- `在线`
- `游戏中`
- `离线`
- `忙碌/处理中` 可选

### 展示位置

- 好友列表头像右下角状态点
- 好友详情标题区文字标签
- 邀请按钮是否可用

### 交互规则

- 在线：可直接邀请
- 游戏中：按钮变为 `观战中/游戏中` 或 `暂不可邀请`
- 离线：按钮改为 `留言邀请` 或禁用

即使第一版后端不支持这么细，也建议前端结构先留出状态位。

## 8. 关键状态设计

好友系统至少应覆盖以下状态：

### 8.1 列表状态

- 初始加载中
- 已加载有数据
- 已加载为空
- 加载失败

### 8.2 搜索状态

- 未输入
- 输入过短
- 搜索中
- 有结果
- 无结果
- 请求失败

### 8.3 申请状态

- 无申请
- 有申请未处理
- 操作提交中
- 已接受
- 已忽略
- 已撤回

### 8.4 邀请状态

- 可邀请
- 已发出邀请等待响应
- 对方接受
- 对方拒绝
- 已过期
- 对方当前不可邀请

这些状态必须体现在 UI 文案和按钮状态上，而不是只在控制台里处理。

## 9. 空态 / 加载 / 异常 方案

## 9.1 空态

### 好友列表为空

建议空态：

- 插图可不做复杂图形，使用简单几何占位 + 小文案
- 文案：`你还没有好友，去搜索一位对手吧。`
- CTA：`去添加好友`

### 申请列表为空

文案：

- `当前没有新的好友申请`

### 搜索无结果

文案：

- `没有找到该用户，试试完整用户名或用户 ID`

## 9.2 Skeleton

建议沿用你当前排行和战绩页的 skeleton 设计语言：

- 蓝色轻微波纹渐变
- 每个好友项一条 skeleton
- 右侧详情区用头像圆形 + 两行文本 + 两个按钮条占位

## 9.3 异常态

错误提示策略：

- 局部失败优先局部提示
- 全局错误才用全局 toast 或对话框

例如：

- 搜索失败：搜索框下方错误文案 + 重试按钮
- 接受申请失败：卡片行内提示
- 删除好友失败：弹窗关闭后 toast 提示

## 10. 微交互与动画规范

结合 `interaction-design`，建议整套好友系统统一使用下列节奏：

### 10.1 动画时长

- hover / pressed：120ms - 150ms
- tab / segment 切换：180ms - 220ms
- 抽屉 / 弹层进入：260ms - 320ms
- 列表插入 / 移除：220ms - 280ms
- toast：240ms 进入，180ms 退出

### 10.2 easing

建议统一：

- 进入：`cubic-bezier(0.16, 1, 0.3, 1)`
- 退出：`cubic-bezier(0.55, 0, 1, 0.45)`

### 10.3 推荐微交互

#### 好友列表项 hover

- 卡片轻微上浮 `translateY(-1px)`
- 边框亮度增强
- 头像外圈 faint glow

#### 添加好友按钮点击

- 按钮轻微缩放到 `0.98`
- 进入 loading 后宽度固定，避免抖动

#### 申请接受成功

- 卡片右侧按钮区域先变成小勾状态 160ms
- 整卡渐隐并折叠

#### 邀请通知进入

- 自上而下滑入 18px + fade-in
- 显示 6-8 秒自动消失
- 悬停时暂停消失计时

## 11. 可访问性与响应式要求

### 11.1 可访问性

- 图标按钮必须带文字或 `aria-label`
- 列表选中态不能只靠颜色，要有边框/激活条/字重变化
- 所有弹层支持 `Esc` 关闭
- 所有关键动作按钮最小高度保持在 40-44px
- 支持 `prefers-reduced-motion`

### 11.2 响应式

#### 桌面端

- 维持双栏结构
- 左栏固定，右栏为详情和操作面板

#### 平板端

- 仍可双栏，但左栏压缩

#### 手机端

- 不建议双栏
- 切为顶部 Tabs + 单栏卡片流
- 好友详情从右侧面板改为底部抽屉或整页详情

## 13. 推荐优先级

## 14. 最终结论

这套好友系统前端方案不应做成一个“新产品”，而应做成现有 KOB 前端里最自然的一块新拼图。

最关键的设计判断有三点：

1. 一级导航新增 `好友`，不藏入口
2. 好友页采用“左列表 + 右详情/申请/发现”的稳态结构
3. PK 页提供“邀请好友”快捷入口，把好友系统真正接回主玩法

UI 方向上，保持你现在已经建立起来的：

- 浅色玻璃感
- 冷蓝主色
- 圆角胶囊交互
- 轻盈短促的动效

交互方向上，重点强化：

- 添加好友的状态连续性
- 接受申请后的反馈闭环
- 好友邀请对战的路径缩短
