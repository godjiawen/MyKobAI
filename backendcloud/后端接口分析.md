# backendcloud 后端接口分析

## 1. 文档范围

这份文档从“接口”视角分析 `backendcloud` 后端，覆盖：

- 主业务服务 `backend` 的 HTTP API
- `backend` 的主对战 WebSocket 协议
- `chatsystem` 的聊天 WebSocket 协议
- `matchingsystem`、`botrunningsystem` 的内部 HTTP 接口
- 每个接口的参数、返回结构、鉴权要求、调用链

说明：

- 这里的接口定义以当前源码实现为准。
- 大部分 HTTP 接口使用 `@RequestParam` 接参，实际更接近 `application/x-www-form-urlencoded` 或普通 query 参数，而不是 JSON Body。
- 返回值多数是 `Map<String, String>`、`JSONObject` 或实体列表，字段结构来自当前实现代码。

## 2. 服务与端口

- `backend`：`3000`
- `matchingsystem`：`3001`
- `botrunningsystem`：`3002`
- `chatsystem`：`3003`

## 3. 鉴权与访问规则

### 3.1 JWT 鉴权

`backend` 的大部分业务接口需要在请求头带：

```http
Authorization: Bearer <token>
```

JWT 的生成入口是：

- `POST /api/user/account/token/`

JWT 中的 `subject` 存的是用户 ID。`backend` 的 `JwtAuthenticationTokenFilter` 会把它解析后写入 `SecurityContextHolder`。

### 3.2 无需登录的接口

`backend` 中明确放开的接口：

- `POST /api/user/account/token/`
- `POST /api/user/account/register/`
- `GET /avatars/**`

### 3.3 仅允许本机访问的内部接口

`backend`：
- `POST /pk/start/game/`
- `POST /pk/receive/bot/move/`

`matchingsystem`：
- `POST /player/add/`
- `POST /player/remove/`

`botrunningsystem`：
- `POST /bot/add/`

这些接口都通过 Spring Security 的 `hasIpAddress('127.0.0.1')` 限制为只能被本机服务调用。

### 3.4 WebSocket 鉴权

- `backend` 的 `/websocket/{token}`：在握手后的 `onOpen()` 中手动解析 token。
- `chatsystem` 的 `/chat/{token}/{roomId}`：同样在 `onOpen()` 中手动解析 token。

也就是说，WebSocket 不走普通 HTTP JWT 过滤器，而是在路径参数中带 token。

## 4. backend HTTP 接口

---

### 4.1 登录接口

#### 接口

- 方法：`POST`
- 路径：`/api/user/account/token/`
- 鉴权：不需要

#### 请求参数

- `username`：用户名
- `password`：密码

#### 返回结构

成功：

```json
{
  "error_message": "success",
  "token": "<jwt>"
}
```

失败：

```json
{
  "error_message": "username or password is incorrect"
}
```

#### 实际调用链

1. `LoginController.getToken()` 接收参数。
2. `LoginServiceImpl.getToken()` 构造 `UsernamePasswordAuthenticationToken`。
3. `AuthenticationManager` 调用 `UserDetailsServiceImpl.loadUserByUsername()` 查用户。
4. 密码校验通过后，调用 `JwtUtil.createJWT(userId)` 生成 token。
5. 返回给前端。

#### 作用

- 登录成功后，这是前端调用其余 HTTP 接口和 WebSocket 的凭证来源。

---

### 4.2 注册接口

#### 接口

- 方法：`POST`
- 路径：`/api/user/account/register/`
- 鉴权：不需要

#### 请求参数

- `username`：用户名
- `password`：密码
- `confirmedPassword`：确认密码

#### 返回结构

成功：

```json
{
  "error_message": "success"
}
```

失败：

```json
{
  "error_message": "<具体校验失败原因>"
}
```

#### 主要校验规则

- 用户名不能为空
- 用户名长度必须在 6 到 12 位之间
- 用户名只允许字母、数字、下划线
- 密码不能为空
- 密码长度必须在 8 到 16 位之间
- 密码不能含空白字符
- 密码复杂度需满足四类字符中至少三类：
- 大写字母
- 小写字母
- 数字
- 特殊字符
- 两次密码必须一致
- 用户名不能重复

#### 返回后的数据效果

- 会向 `user` 表插入一条新用户
- 默认头像使用固定 URL
- 初始积分为 `1500`
- 密码使用 BCrypt 加密保存

#### 实际调用链

1. `RegisterController.register()` 取参数。
2. `RegisterServiceImpl.register()` 做校验。
3. `UserMapper.selectList()` 检查用户名是否存在。
4. `PasswordEncoder.encode()` 加密密码。
5. `UserMapper.insert()` 写入数据库。

### 4.3 获取当前用户信息

#### 接口

- 方法：`GET`
- 路径：`/api/user/account/info/`
- 鉴权：需要 JWT

#### 请求参数

- 无

#### 返回结构

成功：

```json
{
  "error_message": "success",
  "id": "用户ID字符串",
  "username": "用户名",
  "photo": "头像URL"
}
```

#### 实际调用链

1. `JwtAuthenticationTokenFilter` 从请求头解析 token。
2. `InfoController.getinfo()` 调用服务层。
3. `InfoServiceImpl.getinfo()` 从 `SecurityContextHolder` 中取出当前用户。
4. 返回当前用户基础信息。

#### 作用

- 页面刷新后恢复登录用户信息
- 顶部头像、昵称展示
- 前端确认当前登录态身份

### 4.4 修改用户名

#### 接口

- 方法：`POST`
- 路径：`/api/user/account/update/username/`
- 鉴权：需要 JWT

#### 请求参数

- `new_username`：新用户名

#### 返回结构

成功：

```json
{
  "error_message": "success"
}
```

失败：

```json
{
  "error_message": "<具体失败原因>"
}
```

#### 主要校验规则

- 新用户名不能为空
- 格式必须符合 6 到 12 位字母/数字/下划线
- 不能与当前用户名相同
- 不能与数据库中其他用户名重复

#### 实际调用链

1. `UpdateAccountController.updateUsername()` 取 `new_username`。
2. `UpdateAccountServiceImpl.updateUsername()` 从认证上下文拿当前用户。
3. 校验格式与唯一性。
4. `UserMapper.selectById()` 取数据库中的用户。
5. 修改 `username` 字段并 `updateById()`。

### 4.5 修改密码

#### 接口

- 方法：`POST`
- 路径：`/api/user/account/update/password/`
- 鉴权：需要 JWT

#### 请求参数

- `old_password`：旧密码
- `new_password`：新密码
- `confirmed_password`：确认新密码

#### 返回结构

成功：

```json
{
  "error_message": "success"
}
```

失败：

```json
{
  "error_message": "<具体失败原因>"
}
```

#### 主要校验规则

- 旧密码不能为空
- 新密码与确认密码不能为空
- 旧密码必须匹配当前登录用户密码
- 新密码长度必须在 8 到 16 位之间
- 新密码不能包含空白字符
- 新密码复杂度至少满足四类中的三类
- 两次新密码必须一致

#### 实际调用链

1. `UpdateAccountController.updatePassword()` 取参数。
2. `UpdateAccountServiceImpl.updatePassword()` 获取当前用户。
3. 通过 `PasswordEncoder.matches()` 验证旧密码。
4. 通过 `PasswordEncoder.encode()` 加密新密码。
5. `UserMapper.updateById()` 更新数据库。

### 4.6 上传头像

#### 接口

- 方法：`POST`
- 路径：`/api/user/account/avatar/upload/`
- 鉴权：需要 JWT
- 内容类型：`multipart/form-data`

#### 请求参数

- `file`：图片文件

#### 支持格式

- `.jpg`
- `.jpeg`
- `.png`

#### 返回结构

成功：

```json
{
  "error_message": "success",
  "photo": "http://<host>:3000/avatars/<uuid>.<suffix>"
}
```

失败：

```json
{
  "error_message": "<失败原因>"
}
```

#### 服务器行为

- 文件保存在 `kob.upload.dir=${user.dir}/uploads/avatars/`
- 文件名会被替换为 UUID，避免重名
- 用户表中的 `photo` 字段会更新成可访问 URL
- 头像资源通过 `/avatars/**` 暴露

#### 实际调用链

1. `UploadAvatarController.uploadAvatar()` 接收 multipart 文件。
2. `UploadAvatarServiceImpl.upload()` 校验文件。
3. 生成 UUID 文件名并落盘。
4. 用 `ServletUriComponentsBuilder` 生成外部访问 URL。
5. 更新用户头像地址。

### 4.7 新增 Bot

#### 接口

- 方法：`POST`
- 路径：`/api/user/bot/add/`
- 鉴权：需要 JWT

#### 请求参数

- `title`：Bot 标题
- `description`：Bot 描述
- `content`：Bot 源码
- `language`：语言，可选；空值时默认 `java`

#### 支持语言

- `java`
- `python`
- `cpp`
- `javascript`

#### 返回结构

成功：

```json
{
  "error_message": "success"
}
```

失败：

```json
{
  "error_message": "<具体失败原因>"
}
```

#### 主要校验规则

- 标题不能为空
- 标题长度不超过 100
- 描述为空时会自动填默认文案
- 描述长度不超过 300
- 源码不能为空
- 源码长度不超过 10000
- 语言必须属于支持列表
- 每个用户最多只能创建 10 个 Bot

#### 实际调用链

1. `AddController.add()` 接参。
2. `AddServiceImpl.add()` 从认证上下文获取当前用户。
3. 规范化语言并校验内容。
4. `BotMapper.selectCount()` 判断当前用户 Bot 数量是否超限。
5. 构造 `Bot` 实体并插入数据库。

### 4.8 获取当前用户 Bot 列表

#### 接口

- 方法：`GET`
- 路径：`/api/user/bot/getlist/`
- 鉴权：需要 JWT

#### 请求参数

- 无

#### 返回结构

返回值是一个 `Bot` 数组，每项结构大致为：

```json
[
  {
    "id": 1,
    "userId": 100,
    "title": "Bot标题",
    "description": "Bot描述",
    "content": "源码",
    "language": "java",
    "createtime": "2026-04-15 20:00:00",

```

#### 实际调用链

1. `GetListController.getList()` 调服务层。
2. `GetListServiceImpl.getList()` 从当前认证用户中取 ID。
3. `BotMapper.selectList()` 按 `user_id` 查询。
4. 返回当前用户自己的全部 Bot。

#### 接口

- 路径：`/api/user/bot/update/`

#### 请求参数

- `bot_id`：Bot ID

#### 返回结构

成功：

```json

  "error_message": "success"

```

失败：

```json
{
```

### 4.14 内部开局接口

#### 接口

- 方法：`POST`
- 路径：`/pk/start/game/`
- 鉴权：仅允许 `127.0.0.1`

#### 请求参数

- `a_id`：A 玩家用户 ID
- `a_bot_id`：A 方使用的 Bot ID，手动玩家通常为 `-1`
- `b_id`：B 玩家用户 ID
- `b_bot_id`：B 方使用的 Bot ID

#### 返回结构

```json
start game success
```

这是纯字符串，不是 JSON。

#### 实际调用链

1. `matchingsystem` 匹配成功后调用该接口。
2. `StartGameController.startGame()` 解析参数。
3. `StartGameServiceImpl.startGame()` 调用 `WebSocketServer.startGame()`。
4. `WebSocketServer.startGame()`：
- 查双方用户和 Bot
- 创建 `Game`
- 生成地图
- 挂到双方在线连接对象上
- 启动游戏线程
- 向双方推送 `start-matching` 事件包

#### 对前端的结果影响

接口本身不直接返回游戏数据给调用方，而是通过 WebSocket 推送给双方玩家。

### 4.15 内部接收 Bot 方向接口

#### 接口

- 方法：`POST`
- 路径：`/pk/receive/bot/move/`
- 鉴权：仅允许 `127.0.0.1`

#### 请求参数

- `user_id`：哪个用户的 Bot 给出了方向
- `direction`：方向值，约定为 `0-3`

#### 返回结构

```json
receive bot move success
```

纯字符串。

#### 实际调用链

1. `botrunningsystem` 中 `Consumer.run()` 运行 Bot 后调用该接口。
2. `ReceiveBotMoveController.receiveBotMove()` 取参数。
3. `ReceiveBotMoveServiceImpl.receiveBotMove()` 找到该用户在线连接与当前 `Game`。
4. 根据该用户是 A 还是 B，写入 `nextStepA` 或 `nextStepB`。
5. `Game.nextStep()` 轮询到双方方向后继续推进对局。

## 5. backend WebSocket 主对战协议

### 5.1 连接地址

```text
ws://<host>:3000/websocket/{token}
```

其中：

- `{token}`：登录接口签发的 JWT

### 5.2 连接建立后的服务器行为

- `WebSocketServer.onOpen()` 解析 token
- 查用户表确认用户存在
- 将当前连接存入全局在线表 `users`
- 如果 token 无效或用户不存在，连接会被主动关闭

### 5.3 客户端发给服务器的事件

---

#### 事件：开始匹配

客户端发送：

```json
{
  "event": "start-matching",
  "bot_id": -1
}
```

字段说明：

- `event`：固定为 `start-matching`
- `bot_id`：本次匹配使用的 Bot ID；如果是手动玩家，通常用 `-1`

服务端处理：

1. `WebSocketServer.onMessage()` 根据 `event` 分发。
2. 调 `startMatching(botId)`。
3. 由主服务调用 `matchingsystem:/player/add/`。

---

#### 事件：停止匹配

客户端发送：

```json
{
  "event": "stop-matching"
}
```

服务端处理：

1. `WebSocketServer.stopMatching()` 被调用。
2. 主服务调用 `matchingsystem:/player/remove/`。

---

#### 事件：手动移动

客户端发送：

```json
{
  "event": "move",
  "direction": 0
}
```

字段说明：

- `direction`：方向 `0-3`

服务端处理：

1. `WebSocketServer.move(direction)` 检查当前是否有游戏、是否暂停。
2. 只有当前用户对应玩家的 `botId == -1` 时，才接受前端方向。
3. 分别写入 `Game.nextStepA` 或 `Game.nextStepB`。

也就是说：

- 手动玩家才能通过这个事件控制移动
- Bot 玩家前端发 move 不会生效

---

#### 事件：暂停游戏

客户端发送：

```json
{
  "event": "pause"
}
```

服务端处理：

1. `WebSocketServer.pause()` 检查当前是否有游戏、是否已暂停。
2. 根据当前用户判断自己是 A 还是 B。
3. `Game.setPaused(true, by)` 更新状态。
4. 向双方广播 `game-paused`。

---

#### 事件：恢复游戏

客户端发送：

```json
{
  "event": "resume"
}
```

服务端处理：

1. `WebSocketServer.resume()` 检查当前是否处于暂停状态。
2. 只有触发暂停的那一方本人才能恢复。
3. 恢复后广播 `game-resumed`。

### 5.4 服务器推给客户端的事件

---

#### 事件：对局开始

服务器推送：

```json
{
  "event": "start-matching",
  "opponent_username": "对手用户名",
  "opponent_photo": "对手头像",
  "game": {
    "a_id": 1,
    "b_id": 2,
    "a_sx": 11,
    "a_sy": 1,
    "b_sx": 1,
    "b_sy": 12,
    "map": [[0,1,0], [1,0,1]]
  },
  "room_id": "1_2"
}
```

字段说明：

- `event`：当前代码中仍然沿用 `start-matching` 这个事件名，但实际语义是“匹配完成，游戏开始”
- `opponent_username`：对手昵称
- `opponent_photo`：对手头像
- `game`：对局基础信息
- `room_id`：聊天房间号，格式是 `min(aId,bId)_max(aId,bId)`

前端通常会据此：

- 初始化地图
- 确定自己和对手位置
- 连接聊天服务
- 切换到对战界面

---

#### 事件：一回合移动结果

服务器推送：

```json
{
  "event": "move",
  "a_direction": 0,
  "b_direction": 2
}
```

字段说明：

- `a_direction`：A 玩家本回合移动方向
- `b_direction`：B 玩家本回合移动方向

该消息由 `Game.sendMove()` 发送，表示本回合双方输入都已确认，前端应同步推进动画和蛇身状态。

---

#### 事件：游戏暂停

服务器推送：

```json
{
  "event": "game-paused",
  "paused_by": 1
}
```

字段说明：

- `paused_by`：触发暂停的用户 ID

---

#### 事件：游戏恢复

服务器推送：

```json
{
  "event": "game-resumed"
}
```

---

#### 事件：游戏结果

服务器推送：

```json
{
  "event": "result",
  "loser": "A"
}
```

`loser` 取值：

- `A`：A 方失败
- `B`：B 方失败
- `all`：双方同时失败，按平局处理

这条消息发送前，服务端已经完成：

- 双方积分更新
- `record` 表写入

## 6. chatsystem WebSocket 聊天协议

### 6.1 连接地址

```text
ws://<host>:3003/chat/{token}/{roomId}
```

字段说明：

- `{token}`：登录获得的 JWT
- `{roomId}`：主服务开始对局时推送的 `room_id`

### 6.2 连接逻辑

- `ChatWebSocketServer.onOpen()` 解析 token，拿到用户 ID
- 把当前连接加入 `rooms[roomId]`
- 同一个房间内的全部连接共享消息广播

### 6.3 客户端发送消息格式

```json
{
  "content": "你好",
  "username": "玩家A"
}
```

字段说明：

- `content`：消息正文
- `username`：前端显示用用户名

注意：

- 服务端会校验 `content` 不为空/不全是空白
- 服务端并不从数据库查用户名，而是直接使用客户端传来的 `username`，为空时兜底为 `Player`

### 6.4 服务端广播格式

```json
{
  "userId": 1,
  "username": "玩家A",
  "content": "你好",
  "timestamp": 1713180000000
}
```

字段说明：

- `userId`：发送者真实用户 ID，由 JWT 解析得到
- `username`：广播显示名
- `content`：消息内容
- `timestamp`：服务端生成的毫秒时间戳

### 6.5 生命周期行为

- 连接断开时，会从房间连接集合中移除
- 如果房间没人了，会直接删除该 `roomId`

## 7. matchingsystem 内部接口

### 7.1 添加匹配玩家

#### 接口

- 方法：`POST`
- 路径：`/player/add/`
- 鉴权：仅允许本机访问

#### 请求参数

- `user_id`：用户 ID
- `rating`：当前积分
- `bot_id`：选择的 Bot ID

#### 返回结构

```json
add player success
```

纯字符串。

#### 实际调用链

1. `backend` 的 `WebSocketServer.startMatching()` 调这个接口。
2. `MatchingController.addPlayer()` 解析参数。
3. `MatchingServiceImpl.addPlayer()` 委托给 `MatchingPool.addPlayer()`。
4. `MatchingPool` 把用户放进内存等待队列。

---

### 7.2 取消匹配

#### 接口

- 方法：`POST`
- 路径：`/player/remove/`
- 鉴权：仅允许本机访问

#### 请求参数

- `user_id`：用户 ID

#### 返回结构

```json
remove player success
```

#### 实际调用链

1. `backend` 的 `WebSocketServer.stopMatching()` 调这个接口。
2. `MatchingController.removePlayer()` 解析参数。
3. `MatchingServiceImpl.removePlayer()` 委托 `MatchingPool.removePlayer()`。
4. 匹配队列中该用户被移除。

### 7.3 匹配成功后的回调行为

这不是 `matchingsystem` 的公开接口，而是它内部线程的结果动作：

1. `MatchingPool.run()` 每秒执行一次。
2. 先给所有排队玩家 `waitingTime + 1`。
3. `checkMatched(a, b)` 根据 `分差 <= 最小等待时间 * 10` 判断两人是否可配。
4. 如果匹配成功，`sendResult(a, b)` 调用 `backend:/pk/start/game/`。

## 8. botrunningsystem 内部接口

### 8.1 提交 Bot 执行任务

#### 接口

- 方法：`POST`
- 路径：`/bot/add/`
- 鉴权：仅允许本机访问

#### 请求参数

- `user_id`：用户 ID
- `bot_code`：Bot 源码
- `input`：当前对局输入串
- `language`：语言，默认 `java`

#### 返回结构

```json
add bot success
```

#### 实际调用链

1. `backend` 中 `Game.sendBotCode(player)` 调这个接口。
2. `BotRunningController.addBot()` 解析参数。
3. `BotRunningServiceImpl.addBot()` 把任务塞进 `BotPool`。
4. `BotPool` 唤醒线程并创建 `Consumer` 执行。
5. `Consumer` 选择语言 Runner 运行源码。
6. 方向结果回调 `backend:/pk/receive/bot/move/`。

### 8.2 `input` 参数的真实格式

Bot 接收到的 `input` 是一个用 `#` 拼接的字符串，格式为：

```text
地图串#自己起点x#自己起点y#(自己历史步骤)#对手起点x#对手起点y#(对手历史步骤)
```

例如：

```text
001010...#11#1#(0123)#1#12#(1230)
```

含义：

- 地图串：13 x 14 地图按行压平后的 0/1 串
- 自己与对手的起始坐标
- 自己与对手的历史移动方向串

Bot 应根据这个输入输出：

- 一个方向整数 `0/1/2/3`

## 9. 关键链路时序说明

### 9.1 从点击“开始匹配”到开局

1. 前端连接 `backend:/websocket/{token}`。
2. 前端发：

```json
{ "event": "start-matching", "bot_id": -1 }
```

3. `backend` 调 `matchingsystem:/player/add/`。
4. `MatchingPool` 匹配成功后调 `backend:/pk/start/game/`。
5. `backend` 创建 `Game`。
6. `backend` 通过 WebSocket 给双方推送：

```json
{
  "event": "start-matching",
  "opponent_username": "...",
  "opponent_photo": "...",
  "game": {...},
  "room_id": "1_2"
}
```

### 9.2 从一回合开始到 Bot 落子完成

1. `Game.nextStep()` 启动本回合。
2. 对 Bot 玩家调用 `botrunningsystem:/bot/add/`。
3. Bot 被编译/执行。
4. `Consumer` 回调 `backend:/pk/receive/bot/move/`。
5. `Game` 轮询收齐 `nextStepA`、`nextStepB`。
6. `Game.judge()` 判定是否合法。
7. 若对局继续，则发：

```json
{
  "event": "move",
  "a_direction": ...,
  "b_direction": ...
}
```

### 9.3 对局结束时

1. `Game` 判定 `loser`。
2. 调 `saveToDatabase()`：
- 更新双方积分
- 写入 `record` 表
3. 向双方推送：

```json
{
  "event": "result",
  "loser": "A/B/all"
}
```

## 10. 前后端联调时最需要注意的点

### 10.1 HTTP 参数命名必须和源码一致

例如：

- 注册接口是 `confirmedPassword`
- 改用户名是 `new_username`
- 改密码是 `old_password` / `new_password` / `confirmed_password`
- 删除 Bot 是 `bot_id`
- 单局详情是 `record_id`

这些字段名如果前端写错，控制器会拿不到值。

### 10.2 WebSocket 事件名不完全语义化

开局成功后服务端推送的事件名仍然是：

```json
"event": "start-matching"
```

但它实际表示的是“匹配成功并开始游戏”。前端处理时需要按当前代码适配，而不要只看字面含义。

### 10.3 头像接口是文件上传，不是普通 JSON

头像上传必须用 `multipart/form-data`，字段名必须是 `file`。

### 10.4 Bot 玩家与手动玩家输入来源不同

- 手动玩家：通过 WebSocket `move` 事件提交方向
- Bot 玩家：由 `Game` 自动调用 `botrunningsystem` 获取方向

### 10.5 战绩详情与战绩列表用途不同

- `/api/record/getlist/`：用于列表页，附带双方头像、用户名、可读结果
- `/api/record/get/`：用于回放页，提供完整单局记录细节

## 11. 总结

如果从接口角度看，`backendcloud` 后端可以概括成三层：

- 对前端公开的账号、Bot、排行榜、战绩 HTTP API
- 对前端公开的主对战 WebSocket 和聊天 WebSocket
- 服务内部使用的匹配接口、开局接口、Bot 回调接口

整个系统的主链路是：

- 登录拿 JWT
- WebSocket 开始匹配
- 匹配服务找到对手
- 主服务建局并推送开局事件
- 游戏线程逐回合驱动
- Bot 执行服务回传方向
- 对局结束后写库并推送结果
- 聊天服务基于 `room_id` 进行房间广播

- 写接口测试用例
