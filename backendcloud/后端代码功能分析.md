# backendcloud 后端代码功能分析

## 1. 项目整体结构

`backendcloud` 是一个基于 Spring Boot 3 + Spring Security + MyBatis-Plus 的多模块后端项目，按职责拆成四个服务：

- `backend`：主业务服务，负责账号、Bot 管理、排行榜、对局记录、实时对战 WebSocket、游戏判定与结算。
- `matchingsystem`：匹配服务，维护等待队列，按分数和等待时间撮合玩家。
- `botrunningsystem`：Bot 执行服务，接收 Bot 代码，在独立线程/子进程中执行，并把方向结果回传给主服务。
- `chatsystem`：对局聊天服务，提供基于房间号的 WebSocket 聊天。

四个模块通过固定端口本地互调：

- `backend` 运行在 `3000`
- `matchingsystem` 运行在 `3001`
- `botrunningsystem` 运行在 `3002`
- `chatsystem` 运行在 `3003`

### 1.1 典型调用链

#### 玩家开始匹配
1. 前端与 `backend` 的 `/websocket/{token}` 建立连接。
2. 前端发送 `start-matching` 事件给 `WebSocketServer`。
3. `WebSocketServer` 调用 `matchingsystem` 的 `/player/add/`。
4. `MatchingPool` 每秒扫描队列，找到可匹配双方后回调 `backend` 的 `/pk/start/game/`。
5. `backend` 创建 `Game` 线程，并通过 WebSocket 把地图、对手信息、聊天房间号推给双方。

#### Bot 自动落子
1. `Game.nextStep()` 为 Bot 玩家组装输入串。
2. `Game.sendBotCode()` 调用 `botrunningsystem` 的 `/bot/add/`。
3. `botrunningsystem` 把任务放进 `BotPool`，由 `Consumer` 选择对应语言的 Runner 执行。
4. Runner 输出方向后，`Consumer` 回调 `backend` 的 `/pk/receive/bot/move/`。
5. `ReceiveBotMoveServiceImpl` 把方向写回当前 `Game` 对象。
6. `Game` 收齐双方方向后完成判定、广播移动、结束时写入战绩并更新积分。

#### 对局聊天
1. 对局开始时，`WebSocketServer.startGame()` 生成 `room_id`。
2. 前端拿着同一个 JWT 与 `room_id` 连接 `chatsystem` 的 `/chat/{token}/{roomId}`。
3. `ChatWebSocketServer` 按 `roomId` 聚合会话，在同房间广播消息。

## 2. 设计特点与实现思路

- 安全模型：HTTP 接口走 JWT 鉴权；匹配、开局、Bot 回传等服务间接口只允许 `127.0.0.1` 访问；WebSocket 握手则在连接建立时手动解析 token。
- 并发模型：匹配和对局都是常驻线程；对局内部用 `ReentrantLock` 保护暂停、下一步方向等共享状态。
- 游戏模型：蛇前 10 步持续增长，之后每 3 步增长 1 次；地图左右对称随机生成，并强制保证出生点连通。
- 持久化模型：`User`、`Bot`、`Record` 三张核心表通过 MyBatis-Plus 的 `BaseMapper` 直接完成 CRUD。
- Bot 运行模型：Java 使用独立 `javac/java` 子进程，Python/JS/C++ 使用统一 `SubprocessRunner` 模板，避免 Bot 代码直接跑在主服务 JVM 内。

## 3. backend 模块逐文件分析

### 3.1 启动入口与基础配置

#### `backend/src/main/java/com/kob/backend/BackendApplication.java`
- 职责：`backend` 主服务启动入口。
- 实现方式：只有一个 `main` 方法，调用 `SpringApplication.run` 启动 Spring 容器，加载控制器、配置、Mapper、WebSocket 端点等全部 Bean。

#### `backend/src/main/java/com/kob/backend/config/CorsConfig.java`
- 职责：手工处理跨域请求。
- 实现方式：
- 作为优先级最高的 `Filter` 执行。
- 从请求头中读取 `Origin` 和 `Access-Control-Request-Headers`，动态回写到响应头。
- 统一放开所有方法，允许携带 Cookie/认证信息。
- 对 `OPTIONS` 预检请求直接返回 `200`，避免进入后续过滤器链。

#### `backend/src/main/java/com/kob/backend/config/MybatisConfig.java`
- 职责：给 MyBatis-Plus 开启分页拦截器。
- 实现方式：
- 注册 `MybatisPlusInterceptor` Bean。
- 增加 `PaginationInnerInterceptor(DbType.MYSQL)`，使排行榜和战绩分页查询可以直接使用 `Page` 对象。

#### `backend/src/main/java/com/kob/backend/config/RestTemplateConfig.java`
- 职责：提供跨服务 HTTP 调用客户端。
- 实现方式：注册 `RestTemplate` Bean，供 `WebSocketServer` 调匹配服务、`Game` 调 Bot 服务等场景复用。

#### `backend/src/main/java/com/kob/backend/config/SecurityConfig.java`
- 职责：定义主服务的安全边界。
- 实现方式：
- 禁用 CSRF，声明 Session 为 `STATELESS`，说明鉴权完全依赖 JWT。
- 放开登录、注册接口和头像静态资源路径 `/avatars/**`。
- 将 `/pk/start/game/` 与 `/pk/receive/bot/move/` 限制为仅 `127.0.0.1` 可访问，确保只能被内部服务调用。
- 放开所有 `OPTIONS` 请求，兼容跨域预检。
- 其他接口默认需要认证。
- 将 `JwtAuthenticationTokenFilter` 插入用户名密码过滤器前，用于把 JWT 解析结果写入 `SecurityContextHolder`。
- 用 `WebSecurityCustomizer` 忽略 `/websocket/**`，因为 WebSocket 的 token 校验不是在这里做，而是在 `WebSocketServer.onOpen()` 中手动做。

#### `backend/src/main/java/com/kob/backend/config/WebConfig.java`
- 职责：暴露头像上传目录为可访问静态资源。
- 实现方式：
- 通过配置项 `kob.upload.dir` 读取头像目录。
- 启动时若目录不存在则创建。
- 把本地目录映射到 `/avatars/**`，这样用户上传头像后可以直接通过 URL 访问。

#### `backend/src/main/java/com/kob/backend/config/WebSocketConfig.java`
- 职责：启用 `@ServerEndpoint` 类型的 WebSocket 端点。
- 实现方式：注册 `ServerEndpointExporter`，让 `WebSocketServer` 能被 Spring 容器识别并发布。

#### `backend/src/main/java/com/kob/backend/config/filter/JwtAuthenticationTokenFilter.java`
- 职责：HTTP 接口 JWT 认证过滤器。
- 实现方式：
- 从 `Authorization` 头读取 `Bearer token`。
- 使用 `JwtUtil.parseJWT()` 解析出用户 ID。
- 通过 `UserMapper` 查询数据库中的用户对象。
- 将用户包装成 `UserDetailsImpl`，再构造 `UsernamePasswordAuthenticationToken` 写入 `SecurityContextHolder`。
- 如果请求没有 token，则直接放行，由后续安全规则决定是否允许访问。

### 3.2 数据访问与实体

#### `backend/src/main/java/com/kob/backend/mapper/UserMapper.java`
- 职责：用户表访问接口。
- 实现方式：继承 `BaseMapper<User>`，获得按主键增删改查、条件查询、分页等标准能力。

#### `backend/src/main/java/com/kob/backend/mapper/BotMapper.java`
- 职责：Bot 表访问接口。
- 实现方式：同样继承 `BaseMapper<Bot>`，被 Bot 管理、开局加载 Bot、权限校验等流程使用。

#### `backend/src/main/java/com/kob/backend/mapper/RecordMapper.java`
- 职责：对局记录表访问接口。
- 实现方式：继承 `BaseMapper<Record>`，被战绩列表、单局详情和对局结束写库逻辑使用。

#### `backend/src/main/java/com/kob/backend/pojo/User.java`
- 职责：用户实体。
- 实现方式：
- 字段包括 `id`、`username`、`password`、`photo`、`rating`。
- 使用 Lombok 生成 getter/setter/构造器。
- `@TableId(type = IdType.AUTO)` 说明主键自增。

#### `backend/src/main/java/com/kob/backend/pojo/Bot.java`
- 职责：Bot 实体。
- 实现方式：
- 保存 Bot 所属用户、标题、描述、源码、语言、创建时间、修改时间。
- `language` 字段配合 `BotLanguageUtil` 支持多语言 Bot。
- 时间字段通过 `@JsonFormat` 固定输出格式和时区，便于前端直接展示。

#### `backend/src/main/java/com/kob/backend/pojo/Record.java`
- 职责：对局记录实体。
- 实现方式：
- 保存双方用户 ID、出生点、行走轨迹、地图串、失败方、创建时间。
- `aSteps` / `bSteps` 保存完整方向序列，前端回放时可以按步重建局面。

### 3.3 认证与通用工具

#### `backend/src/main/java/com/kob/backend/utils/JwtUtil.java`
- 职责：生成和解析 JWT。
- 实现方式：
- `createJWT(subject)` 以用户 ID 作为 `subject` 生成 token。
- 有效期设置为 14 天。
- `generalKey()` 从固定字符串做 Base64 解码后构造 HMAC 密钥。
- `parseJWT()` 返回 Claims，供过滤器和 WebSocket 鉴权复用。

#### `backend/src/main/java/com/kob/backend/utils/BotLanguageUtil.java`
- 职责：Bot 语言规范化与白名单校验。
- 实现方式：
- 支持 `java`、`python`、`cpp`、`javascript` 四种语言。
- `normalize()` 统一做去空格和小写化，空值默认 `java`。
- `isSupported()` 基于白名单判断语言是否合法。

### 3.4 Spring Security 用户封装

#### `backend/src/main/java/com/kob/backend/service/impl/utils/UserDetailsImpl.java`
- 职责：把项目自己的 `User` 实体接入 Spring Security。
- 实现方式：
- 持有一个 `User user`。
- `getUsername()`、`getPassword()` 都直接透传实体字段。
- 不做角色权限划分，所以 `getAuthorities()` 返回 `null`。
- 账号状态四个布尔方法全部返回 `true`，表示未实现冻结、过期、锁定等高级状态。

#### `backend/src/main/java/com/kob/backend/service/impl/UserDetailsServiceImpl.java`
- 职责：登录时按用户名加载用户。
- 实现方式：
- 用 `UserMapper` + `QueryWrapper` 按 `username` 精确查询。
- 查到用户就包装成 `UserDetailsImpl` 返回给 `AuthenticationManager`。
- 查不到直接抛异常，登录流程据此判定失败。

### 3.5 账号接口与实现

#### `backend/src/main/java/com/kob/backend/controller/user/account/LoginController.java`
- 职责：登录接口入口，路由 `/api/user/account/token/`。
- 实现方式：从表单参数中取 `username`、`password`，交给 `LoginService`。

#### `backend/src/main/java/com/kob/backend/service/user/account/LoginService.java`
- 职责：登录服务接口定义。
- 实现方式：只声明 `getToken` 方法，解耦控制器与实现。

#### `backend/src/main/java/com/kob/backend/service/impl/user/account/LoginServiceImpl.java`
- 职责：完成用户名密码认证并签发 JWT。
- 实现方式：
- 构造 `UsernamePasswordAuthenticationToken` 交给 `AuthenticationManager`。
- 认证成功后，从 `UserDetailsImpl` 中拿到用户 ID，调用 `JwtUtil.createJWT()` 生成 token。
- 返回 `{error_message, token}`。
- 认证异常统一返回“用户名或密码错误”。

#### `backend/src/main/java/com/kob/backend/controller/user/account/RegisterController.java`
- 职责：注册接口入口，路由 `/api/user/account/register/`。
- 实现方式：提取 `username`、`password`、`confirmedPassword`，交给注册服务。

#### `backend/src/main/java/com/kob/backend/service/user/account/RegisterService.java`
- 职责：注册服务接口定义。
- 实现方式：约定注册参数和返回值格式。

#### `backend/src/main/java/com/kob/backend/service/impl/user/account/RegisterServiceImpl.java`
- 职责：创建新用户，并完成用户名/密码校验。
- 实现方式：
- 用户名必须是 6 到 12 位，仅允许字母、数字、下划线。
- 密码必须是 8 到 16 位，不能包含空白字符，且必须满足“大写/小写/数字/特殊字符”四类里至少三类。
- 确认密码必须一致。
- 通过 `UserMapper` 检查用户名唯一性。
- 用 `BCryptPasswordEncoder` 加密密码。
- 给新用户设置默认头像与初始积分 `1500`，再插入数据库。

#### `backend/src/main/java/com/kob/backend/controller/user/account/InfoController.java`
- 职责：获取当前登录用户信息，路由 `/api/user/account/info/`。
- 实现方式：不接收参数，直接调用 `InfoService`。

#### `backend/src/main/java/com/kob/backend/service/user/account/InfoService.java`
- 职责：当前用户信息服务接口。

#### `backend/src/main/java/com/kob/backend/service/impl/user/account/InfoServiceImpl.java`
- 职责：从当前认证上下文中取出用户基础资料。
- 实现方式：
- 从 `SecurityContextHolder` 取得 `UsernamePasswordAuthenticationToken`。
- 强转 principal 为 `UserDetailsImpl`。
- 返回用户 `id`、`username`、`photo`，同时附带 `error_message=success`。

#### `backend/src/main/java/com/kob/backend/controller/user/account/UpdateAccountController.java`
- 职责：账号资料修改入口。
- 实现方式：
- `/api/user/account/update/username/`：修改用户名。
- `/api/user/account/update/password/`：修改密码。
- 仅负责取参数并转给服务层。

#### `backend/src/main/java/com/kob/backend/service/user/account/UpdateAccountService.java`
- 职责：账号修改服务接口，拆成改用户名和改密码两个方法。

#### `backend/src/main/java/com/kob/backend/service/impl/user/account/UpdateAccountServiceImpl.java`
- 职责：处理用户名和密码更新。
- 实现方式：
- 先从安全上下文中拿到当前登录用户。
- 改用户名时：
- 校验非空、格式、唯一性。
- 禁止改成与当前用户名相同的值。
- 从数据库重新查出用户后更新并写回。
- 改密码时：
- 校验旧密码存在且与当前加密密码匹配。
- 新密码继续沿用注册时的长度、空白字符、复杂度规则。
- 确认密码一致后用 `PasswordEncoder` 重新加密并更新数据库。

#### `backend/src/main/java/com/kob/backend/controller/user/account/UploadAvatarController.java`
- 职责：头像上传入口，路由 `/api/user/account/avatar/upload/`。
- 实现方式：接收 `MultipartFile file`，转交 `UploadAvatarService`。

#### `backend/src/main/java/com/kob/backend/service/user/account/UploadAvatarService.java`
- 职责：头像上传服务接口。

#### `backend/src/main/java/com/kob/backend/service/impl/user/account/UploadAvatarServiceImpl.java`
- 职责：保存头像文件并更新用户头像 URL。
- 实现方式：
- 校验文件不为空。
- 从安全上下文拿到当前用户。
- 校验原始文件名与后缀，仅允许 `.jpg`、`.jpeg`、`.png`。
- 用 UUID 生成新文件名，保存在 `kob.upload.dir` 指定目录下。
- 通过 `ServletUriComponentsBuilder.fromCurrentContextPath()` 拼出可访问 URL。
- 更新数据库中的 `photo` 字段，并把新 URL 返回给前端。

### 3.6 Bot 管理接口与实现

#### `backend/src/main/java/com/kob/backend/controller/user/bot/AddController.java`
- 职责：新增 Bot 接口入口，路由 `/api/user/bot/add/`。
- 实现方式：直接把参数 Map 传给 `AddService`。

#### `backend/src/main/java/com/kob/backend/service/user/bot/AddService.java`
- 职责：新增 Bot 服务接口。

#### `backend/src/main/java/com/kob/backend/service/impl/user/bot/AddServiceImpl.java`
- 职责：为当前用户创建 Bot。
- 实现方式：
- 从登录上下文中识别当前用户。
- 读取标题、描述、源码、语言；语言先用 `BotLanguageUtil.normalize()` 标准化。
- 校验标题非空且不超过 100 字。
- 描述为空时给默认文案，最长 300 字。
- 源码不能为空且不超过 10000 字。
- 语言必须在支持列表中。
- 限制每个用户最多 10 个 Bot。
- 创建 `Bot` 实体，写入创建/修改时间后插入数据库。

#### `backend/src/main/java/com/kob/backend/controller/user/bot/GetListController.java`
- 职责：获取当前用户 Bot 列表，路由 `/api/user/bot/getlist/`。
- 实现方式：无参数，调用 `GetListService`。

#### `backend/src/main/java/com/kob/backend/service/user/bot/GetListService.java`
- 职责：Bot 列表服务接口。

#### `backend/src/main/java/com/kob/backend/service/impl/user/bot/GetListServiceImpl.java`
- 职责：查询当前登录用户的全部 Bot。
- 实现方式：
- 从安全上下文中获取用户 ID。
- 使用 `BotMapper` 按 `user_id` 条件查询并返回列表。

#### `backend/src/main/java/com/kob/backend/controller/user/bot/RemoveController.java`
- 职责：删除 Bot 接口入口，路由 `/api/user/bot/remove/`。

#### `backend/src/main/java/com/kob/backend/service/user/bot/RemoveService.java`
- 职责：删除 Bot 服务接口。

#### `backend/src/main/java/com/kob/backend/service/impl/user/bot/RemoveServiceImpl.java`
- 职责：删除用户自己的 Bot。
- 实现方式：
- 根据 `bot_id` 查数据库。
- 如果 Bot 不存在直接报错。
- 如果 Bot 不属于当前登录用户则拒绝删除。
- 校验通过后执行 `deleteById`。

#### `backend/src/main/java/com/kob/backend/controller/user/bot/UpdateController.java`
- 职责：更新 Bot 接口入口，路由 `/api/user/bot/update/`。

#### `backend/src/main/java/com/kob/backend/service/user/bot/UpdateService.java`
- 职责：更新 Bot 服务接口。

#### `backend/src/main/java/com/kob/backend/service/impl/user/bot/UpdateServiceImpl.java`
- 职责：修改现有 Bot。
- 实现方式：
- 先读取 `bot_id` 和新的标题、描述、源码、语言。
- 校验规则与新增 Bot 基本一致。
- 查询旧 Bot 并校验归属权。
- 创建一个新的 `Bot` 对象：保留旧 `id`、`userId`、`createtime`，只替换可修改字段，并用当前时间覆盖 `modifytime`。
- 调用 `updateById` 完成更新。

### 3.7 排行榜与战绩接口

#### `backend/src/main/java/com/kob/backend/controller/ranklist/GetRanklistController.java`
- 职责：排行榜接口入口，路由 `/api/ranklist/getlist/`。
- 实现方式：从请求参数中读取页码 `page`，转给服务层。

#### `backend/src/main/java/com/kob/backend/service/ranklist/GetRanklistService.java`
- 职责：排行榜服务接口。

#### `backend/src/main/java/com/kob/backend/service/impl/ranklist/GetRanklistServiceImpl.java`
- 职责：按积分分页返回排行榜。
- 实现方式：
- 使用 `Page<>(page, 10)` 固定每页 10 条。
- `QueryWrapper` 按 `rating` 倒序。
- 查询后把每个用户的密码字段清空，防止返回给前端。
- 最终返回 `users` 和 `users_count`。

#### `backend/src/main/java/com/kob/backend/controller/record/GetRecordListController.java`
- 职责：战绩列表接口入口，路由 `/api/record/getlist/`。

#### `backend/src/main/java/com/kob/backend/service/record/GetRecordListService.java`
- 职责：战绩列表服务接口。

#### `backend/src/main/java/com/kob/backend/service/impl/record/GetRecordListServiceImpl.java`
- 职责：分页返回对局记录，并把双方用户信息组装好。
- 实现方式：
- 按记录 ID 倒序分页，每页 10 条。
- 对每条 `Record`，分别查询 A/B 玩家资料。
- 额外拼接 `a_photo`、`a_username`、`b_photo`、`b_username`。
- 根据 `loser` 字段翻译出本局结果文本。
- 用户被删除时，用户名退化为占位文案，避免接口报错。
- 返回 `records` 和 `records_count`。

#### `backend/src/main/java/com/kob/backend/controller/record/GetRecordController.java`
- 职责：单局详情接口入口，路由 `/api/record/get/`。

- `getCells()` 根据出生点与历史步骤重建当前整条蛇的身体坐标。
- `getStepsString()` 把方向列表编码成字符串，便于持久化和传给 Bot。

#### `backend/src/main/java/com/kob/backend/consumer/utils/JwtAuthentication.java`
- 职责：给 WebSocket 层提供简化版 JWT 用户解析。
- 实现方式：直接调用 `JwtUtil.parseJWT()`，取出 `subject` 并转成用户 ID。

#### `backend/src/main/java/com/kob/backend/consumer/utils/Game.java`
- 职责：对局核心线程，负责地图生成、回合推进、Bot 调用、碰撞判定、积分结算、战绩落库。
- 实现方式：
- 构造阶段：
- 保存地图尺寸 `13x14`、内部墙数量 `20`。
- 根据双方 Bot 是否为空，决定 `botId=-1` 代表手动玩家，或保存 Bot 代码/语言供后续调用。
- 固定双方出生点：A 在左下、B 在右上。
- 地图生成：
- `draw()` 先清空地图并构造外边框墙。
- 随机生成内部墙，且关于中心对称，保证双方公平。
- 避开双方出生点。
- `check_connectivity()` 用 DFS 验证出生点之间可达，防止出现无解地图。
- `createMap()` 最多尝试 1000 次直到生成可用地图。
- Bot 输入组织：
- `getMapString()` 把地图压平成由 `0/1` 组成的字符串。
- `getInput(player)` 按“地图#自己起点#自己步骤#对手起点#对手步骤”的格式组装输入串。
- 回合推进：
- `sendBotCode()` 仅对有 Bot 的玩家调用 `botrunningsystem`。
- `nextStep()` 每回合先短暂休眠，再发起 Bot 请求，然后轮询等待双方方向。
- 轮询时处理暂停状态；如果在超时时间内同时收到 `nextStepA` 和 `nextStepB`，就把方向加入双方步骤列表。
- 判定逻辑：
- `check_valid()` 检查蛇头是否撞墙、撞自己、撞对手。
- `judge()` 分别检查双方是否合法，决定 `status` 和 `loser`。
- 广播逻辑：
- `sendMove()` 广播本回合双方方向，并把 `nextStepA/B` 清空，为下一回合做准备。
- `sendResult()` 广播最终结果。
- 数据持久化：
- `saveToDatabase()` 根据输赢更新双方积分：输家 `-2`，赢家 `+5`，平局双方 `+1`。
- 构造 `Record` 写入数据库，记录完整路径与地图。
- 线程主循环：
- `run()` 最多跑 1000 回合。
- 每回合成功拿到方向就继续推进；若超时，则按未返回方向的一方判负或双方都没返回判平。
- 结束后广播结果并终止线程。

#### `backend/src/main/java/com/kob/backend/consumer/WebSocketServer.java`
- 职责：主对战 WebSocket 端点，是前端实时交互的核心入口。
- 实现方式：
- 连接管理：
- 端点地址为 `/websocket/{token}`。
- `onOpen()` 手动解析 JWT，查用户表，验证通过后把当前连接放入 `users` 这个 `ConcurrentHashMap<Integer, WebSocketServer>`。
- `onClose()` 断开时把用户从在线表移除。
- Bean 注入：
- 由于 `@ServerEndpoint` 不是普通 Spring MVC Bean，代码用 `@Autowired` setter 把 `UserMapper`、`RecordMapper`、`BotMapper`、`RestTemplate` 写入静态字段，供静态方法和游戏线程访问。
- 匹配控制：
- `startMatching(botId)` 调匹配服务 `/player/add/`，把当前用户 ID、积分和 Bot ID 发过去。
- `stopMatching()` 调匹配服务 `/player/remove/` 把自己从等待队列删除。
- 开局逻辑：
- `startGame(aId, aBotId, bId, bBotId)` 是全局静态方法，由匹配服务回调触发。
- 查询双方用户和 Bot。
- 创建 `Game`，生成地图并挂到双方的 `WebSocketServer.game` 字段上。
- 启动 `Game` 线程。
- 组装对手信息、出生点、地图、聊天房间号 `room_id=minId_maxId`，分别推送给双方前端。
- 对局内事件：
- `move(direction)`：只允许手动玩家提交移动；如果该位置是 Bot，则忽略前端移动。
- `pause()`：把对局标记为暂停，并广播 `game-paused`。
- `resume()`：只有发起暂停的一方自己才能恢复，并广播 `game-resumed`。
- `broadcastToGame()`：向当前对局双方同时发消息。
- 路由分发：
- `onMessage()` 根据 `event` 字段分发到 `start-matching`、`stop-matching`、`move`、`pause`、`resume`。
- 发送消息：
- `sendMessage()` 对 `session` 加同步锁，避免并发写 WebSocket 出现问题。

- 职责：匹配服务实现。
- 实现方式：
- 持有全局静态 `MatchingPool`。
- `addPlayer()` / `removePlayer()` 只负责打印日志并把操作委托给线程池对象。

- 职责：匹配队列里的玩家模型。
- 实现方式：保存用户 ID、积分、所选 Bot ID、等待时间。

#### `matchingsystem/src/main/java/com/kob/matchingsystem/service/impl/utils/MatchingPool.java`
- 职责：匹配核心线程。
- 实现方式：
- 用 `List<Player>` 保存等待队列，用 `ReentrantLock` 保护并发访问。

- 未匹配成功的玩家会被保留到新列表中，等待下一轮。
- `run()` 每秒执行一次等待时间增长与匹配扫描。

- 实现方式：仅允许本机访问 `/bot/add/`，其余请求默认需要认证。

#### `botrunningsystem/src/main/java/com/kob/botrunningsystem/controller/BotRunningController.java`
- 职责：Bot 任务入口，路由 `/bot/add/`。
- 实现方式：
- 接收 `user_id`、`bot_code`、`input`、`language`。
- `language` 参数为空时默认 `java`。
- 把数据转给 `BotRunningService`。

#### `botrunningsystem/src/main/java/com/kob/botrunningsystem/service/BotRunningService.java`
- 职责：Bot 执行服务接口。

#### `botrunningsystem/src/main/java/com/kob/botrunningsystem/service/impl/BotRunningServiceImpl.java`
- 职责：把 Bot 执行请求放入队列。
- 实现方式：
- 持有全局静态 `BotPool`。
- `addBot()` 记录日志后委托 `botpool.addBot()`。

#### `botrunningsystem/src/main/java/com/kob/botrunningsystem/service/impl/utils/Bot.java`
- 职责：Bot 运行任务模型。
- 实现方式：保存用户 ID、源代码、输入串、语言类型，供队列消费。

#### `botrunningsystem/src/main/java/com/kob/botrunningsystem/service/impl/utils/BotPool.java`
- 职责：Bot 任务消费队列线程。
- 实现方式：
- 内部是 `Queue<Bot>`，配合 `ReentrantLock + Condition` 实现生产者/消费者模型。
- `addBot()` 入队后 `signalAll()` 唤醒等待线程。
- `run()` 在队列为空时阻塞等待；拿到任务后在锁外调用 `consume()`，避免长时间执行 Bot 时阻塞新任务入队。
- `consume()` 为每个任务创建 `Consumer` 并设定 2 秒超时。

#### `botrunningsystem/src/main/java/com/kob/botrunningsystem/service/impl/utils/Consumer.java`
- 职责：真正执行单个 Bot 并回传方向。
- 实现方式：
- `startTimeout()` 启动线程后 `join(timeout)`，超时后主动 `interrupt()`，作为线程级兜底超时控制。
- `getRunner(language)` 根据语言选择 `JavaRunner`、`PythonRunner`、`CppRunner`、`JsRunner`。
- `run()` 内部调用对应 Runner 执行代码，异常时方向默认回退为 `0`。
- 最后通过 HTTP 回调主服务 `/pk/receive/bot/move/`。

#### `botrunningsystem/src/main/java/com/kob/botrunningsystem/service/impl/utils/runners/LanguageRunner.java`
- 职责：所有语言运行器的统一接口。
- 实现方式：约定 `run(code, input, timeoutMs)` 返回方向 `0-3`，出错时约定返回 `0`。

#### `botrunningsystem/src/main/java/com/kob/botrunningsystem/service/impl/utils/runners/SubprocessRunner.java`
- 职责：Python/JS/C++ 这类“写文件后拉起子进程”的通用运行模板。
- 实现方式：
- 为每次执行创建独立临时目录。
- 写入 `input.txt` 和源码文件 `bot.xxx`。
- 子类只需要提供扩展名和 `ProcessBuilder`。
- 启动子进程后，用两个守护线程异步读取 stdout/stderr，避免缓冲区堵塞。
- 在给定超时内等待进程结束，超时则强杀并返回 `0`。
- 只读取第一行 stdout 作为方向值，并校验必须在 `0-3` 之间。
- 结束后递归删除临时目录。

#### `botrunningsystem/src/main/java/com/kob/botrunningsystem/service/impl/utils/runners/PythonRunner.java`
- 职责：Python Bot 运行器。
- 实现方式：
- 继承 `SubprocessRunner`。
- Windows 下执行命令 `python bot.py`，非 Windows 使用 `python3 bot.py`。

#### `botrunningsystem/src/main/java/com/kob/botrunningsystem/service/impl/utils/runners/JsRunner.java`
- 职责：JavaScript Bot 运行器。
- 实现方式：继承 `SubprocessRunner`，用 `node bot.js` 运行。

#### `botrunningsystem/src/main/java/com/kob/botrunningsystem/service/impl/utils/runners/CppRunner.java`
- 职责：C++ Bot 运行器。
- 实现方式：
- 先在临时目录中执行 `g++ -O2 -o bot bot.cpp` 编译。
- 编译阶段单独限制 10 秒，超时或失败直接抛异常。
- 编译成功后再运行生成的可执行文件，执行阶段由父类控制超时。
- Windows 输出为 `bot.exe`，其他平台为 `./bot`。

#### `botrunningsystem/src/main/java/com/kob/botrunningsystem/service/impl/utils/runners/JavaRunner.java`
- 职责：Java Bot 运行器。
- 实现方式：
- 不直接把用户代码塞进当前 JVM，而是写入临时目录并用 `javac/java` 独立编译运行，安全性和隔离性比动态加载类更好。
- 要求用户代码里必须存在 `public class Bot`。
- 自动给类名拼一个随机后缀，避免类名冲突。
- 自动包装进固定包名 `com.kob.botrunningsystem.utils`，再生成一个 `RunnerMain` 作为统一入口。
- 编译成功后执行 `java -cp 临时目录 RunnerMain`。
- 解析第一行输出作为方向；若超时、编译失败、输出非法，则返回 `0`。
- 最后清理临时目录。

#### `botrunningsystem/src/main/java/com/kob/botrunningsystem/utils/BotInterface.java`
- 职责：示例 Bot 约定接口。
- 实现方式：只定义 `nextMove(String input)`，用于说明 Bot 代码应具备的行为接口。

#### `botrunningsystem/src/main/java/com/kob/botrunningsystem/utils/Bot.java`
- 职责：示例/模板 Bot。
- 实现方式：
- 读取 `input.txt`。
- 把地图、双方位置、双方路径解析出来。
- 用和主服务相同的蛇身增长规则重建占用格。
- 从四个方向里选择第一个可走的合法方向返回。
- 该类实现 `Supplier<Integer>`，便于作为 Bot 代码结构的参考。

- 职责：聊天服务内部的 JWT 解析工具。
- 实现方式：
- 使用与主服务完全相同的 `JWT_KEY`，保证同一 token 可以跨服务识别。

- 职责：对局聊天 WebSocket 端点，地址为 `/chat/{token}/{roomId}`。
- 实现方式：
- 使用 `ConcurrentHashMap<String, CopyOnWriteArraySet<ChatWebSocketServer>> rooms` 保存房间到连接集合的映射。
- `onOpen()`：
- 解析路径中的 JWT token，失败就关闭连接。
- 记录当前用户 ID 与房间号。
- 把连接加入房间集合。

- 退出当前房间。
- 如果房间空了就从全局 Map 删除。
- `onMessage()`：
- 把消息解析成 JSON。
- 读取 `content` 与 `username`。
- 过滤空消息。
- 组装广播包，包含发送者 `userId`、`username`、`content`、`timestamp`。
- 向该房间所有连接广播。
- `sendMessage()`：对 session 加同步锁，保证并发发送安全。

- `/pk/receive/bot/move/`：内部 Bot 结果回调。

### 7.2 WebSocket 接口
- `backend:/websocket/{token}`：对战主通道，处理匹配、移动、暂停、恢复。
- `chatsystem:/chat/{token}/{roomId}`：对局聊天通道。

### 7.3 服务间调用关系
- `backend -> matchingsystem`：开始匹配、取消匹配。
- `matchingsystem -> backend`：匹配成功后通知开局。
- `backend -> botrunningsystem`：请求执行 Bot 代码。
- `botrunningsystem -> backend`：回传 Bot 方向。
- `backend -> chatsystem`：没有直接 HTTP 调用，但会生成双方共用的 `room_id`，让前端连接聊天服务。

## 8. 代码层面的优点与注意点

### 8.1 优点
- 模块边界清楚，匹配、Bot 执行、聊天都从主业务中拆了出去。
- HTTP 鉴权、WebSocket 鉴权、服务间本机访问限制三层边界比较明确。
- `Game` 对地图生成、判定、持久化、广播做了完整闭环，实现上是自洽的。
- Bot 多语言支持不是只停留在字段层，而是已经做到了执行层。

### 8.2 需要注意的点
- `WebSocketServer` 通过静态字段持有 Mapper 和 `RestTemplate`，写起来方便，但测试和扩展性一般。
- `MatchingPool` 和 `BotPool` 都是单机内存队列/线程模型，适合当前项目，不适合多实例横向扩展。
- Bot 运行已做进程隔离和超时控制，但还没有更强的资源沙箱，例如 CPU/内存/系统调用级别限制。
- `UserDetailsImpl.getAuthorities()` 返回 `null`，当前系统没有角色权限体系，后续如果做管理后台需要补角色模型。
- 测试文件较少，目前主要依赖运行时验证，缺少核心逻辑的自动化测试。

## 9. 总结

从代码实现上看，这个后端不是简单的 CRUD 项目，而是围绕“联机蛇对战”做了完整的后端闭环：

- `backend` 管业务、鉴权、WebSocket 对战与结算。
- `matchingsystem` 管排队和撮合。
- `botrunningsystem` 管 Bot 的多语言执行与结果回传。
- `chatsystem` 管对局内聊天。

如果从“每个 Java 文件分别负责什么、如何实现”的角度看，这套代码的核心脉络可以概括为：

- Controller 负责收参数。
- Service 接口定义业务边界。
- ServiceImpl 负责真正业务逻辑。
- Mapper 负责数据访问。
- WebSocketServer 和 Game 负责实时对战。
- MatchingPool 和 BotPool 负责后台异步线程。
- 各 Runner 负责多语言 Bot 执行细节。
