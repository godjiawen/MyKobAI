# backendcloud 前端联调用接口清单

## 1. 文档用途

这份文档不是从后端模块出发，而是从前端联调出发整理接口清单，适合以下场景：

- 前端按页面接接口
- 联调时快速确认请求方式、参数名、鉴权要求
- 排查“为什么接口没返回数据”“为什么 WebSocket 没反应”
- 对照页面功能检查接口是否接全

文档按常见前端页面/功能区分组：

- 登录与注册页
- 顶部用户信息与个人资料
- Bot 管理页
- 排行榜页
- 对局记录页
- 匹配与对战页
- 对局聊天功能

## 2. 联调前统一约定

### 2.1 服务地址

默认端口：

- `backend`：`http://127.0.0.1:3000`
- `matchingsystem`：`http://127.0.0.1:3001`
- `botrunningsystem`：`http://127.0.0.1:3002`
- `chatsystem`：`ws://127.0.0.1:3003`

前端实际直接调的主要是：

- `backend` 的 HTTP 接口
- `backend` 的主 WebSocket
- `chatsystem` 的聊天 WebSocket

### 2.2 JWT 使用方式

登录成功后，前端会拿到：

```json
{
  "error_message": "success",
  "token": "<jwt>"
}
```

后续所有需要登录的 HTTP 接口都要带请求头：

```http
Authorization: Bearer <token>
```

注意：

- `Bearer` 后面有空格
- 如果不带这个头，大部分接口会被 Spring Security 拦掉

### 2.3 WebSocket 使用方式

两个 WebSocket 都是把 token 放到路径里，不是在 header 里传：

- 主对战 WebSocket：`ws://127.0.0.1:3000/websocket/{token}`
- 聊天 WebSocket：`ws://127.0.0.1:3003/chat/{token}/{roomId}`

### 2.4 参数提交方式

源码中控制器大多用 `@RequestParam` 接参，所以前端联调时建议：

- GET：用 query 参数
- POST：优先用 `application/x-www-form-urlencoded`
- 上传头像：用 `multipart/form-data`

不要默认按 JSON body 去传，除非你自己确认过封装层已经把它转成了表单格式。

### 2.5 返回结构约定

这个项目没有统一的全局响应包装类，所以返回结构分三类：

- `Map<String, String>`：常见于账号、Bot 管理
- `JSONObject`：常见于排行榜、战绩、单局详情
- 实体数组：常见于 Bot 列表
- 纯字符串：常见于内部接口

前端处理时不要假设所有接口都有 `code`、`msg`、`data` 这类统一结构。

## 3. 登录与注册页

---

### 3.1 登录

#### 适用页面

- 登录页
- 登录弹窗

#### 接口

- 方法：`POST`
- 地址：`/api/user/account/token/`
- 鉴权：不需要

#### 请求参数

- `username`
- `password`

#### 请求示例

```text
username=test_user&password=Test@123
```

#### 返回示例

成功：

```json
{
  "error_message": "success",
  "token": "<jwt>"
}
```

失败：

```json
{
  "error_message": "username or password is incorrect"
}
```

#### 前端联调要点

- 登录成功后要把 `token` 持久化，通常放到 localStorage 或状态管理里。
- 后续请求和 WebSocket 连接都依赖这个 token。
- 如果登录失败，直接展示 `error_message` 即可。

#### 页面建议动作

- 保存 token
- 拉取 `/api/user/account/info/` 恢复当前用户信息
- 跳转首页或大厅页

---

### 3.2 注册

#### 适用页面

- 注册页
- 登录页中的注册弹窗

#### 接口

- 方法：`POST`
- 地址：`/api/user/account/register/`
- 鉴权：不需要

#### 请求参数

- `username`
- `password`
- `confirmedPassword`

#### 请求示例

```text
username=test_01&password=Test@123&confirmedPassword=Test@123
```

#### 返回示例

成功：

```json
{
  "error_message": "success"
}
```

失败：

```json
{
  "error_message": "具体错误原因"
}
```

#### 前端联调要点

- 参数名必须是 `confirmedPassword`，大小写不能改。
- 建议前端先做一次基础校验，再调用后端，减少无效请求。
- 后端已经有完整校验，失败时直接用 `error_message` 提示用户即可。

#### 常见失败原因

- 用户名重复
- 用户名长度/格式不符合要求
- 密码长度不合规
- 密码复杂度不够
- 两次密码不一致

## 4. 顶部用户信息与个人资料页

---

### 4.1 获取当前登录用户信息

#### 适用页面

- 顶部导航栏
- 个人中心页
- 应用初始化时恢复登录态

#### 接口

- 方法：`GET`
- 地址：`/api/user/account/info/`
- 鉴权：需要 JWT

#### 请求头

```http
Authorization: Bearer <token>
```

#### 返回示例

```json
{
  "error_message": "success",
  "id": "1",
  "username": "test_user",
  "photo": "http://127.0.0.1:3000/avatars/xxx.png"
}
```

#### 前端联调要点

- 一般在应用启动时调用，用于恢复登录态。
- 如果 token 失效，接口会拿不到正常结果，前端应清空本地 token 并跳回登录页。
- `id` 返回的是字符串，不是数字，前端如果依赖数字比较要注意转换。

---

### 4.2 修改用户名

#### 适用页面

- 个人资料设置页

#### 接口

- 方法：`POST`
- 地址：`/api/user/account/update/username/`
- 鉴权：需要 JWT

#### 请求参数

- `new_username`

#### 返回示例

成功：

```json
{
  "error_message": "success"
}
```

失败：

```json
{
  "error_message": "具体错误原因"
}
```

#### 前端联调要点

- 参数名必须是 `new_username`。
- 修改成功后，建议重新调一次 `/api/user/account/info/` 刷新顶部显示。
- 后端会拦截“新用户名和旧用户名相同”的情况。

---

### 4.3 修改密码

#### 适用页面

- 安全设置页
- 个人中心修改密码区

#### 接口

- 方法：`POST`
- 地址：`/api/user/account/update/password/`
- 鉴权：需要 JWT

#### 请求参数

- `old_password`
- `new_password`
- `confirmed_password`

#### 返回示例

成功：

```json
{
  "error_message": "success"
}
```

失败：

```json
{
  "error_message": "具体错误原因"
}
```

#### 前端联调要点

- 参数名必须是 `confirmed_password`，不是 `confirmedPassword`。
- 修改成功后，前端可以提示用户重新登录，虽然当前代码并不会立刻让旧 token 失效，但业务上这样更稳妥。

---

### 4.4 上传头像

#### 适用页面

- 个人资料页
- 顶部头像编辑功能

#### 接口

- 方法：`POST`
- 地址：`/api/user/account/avatar/upload/`
- 鉴权：需要 JWT
- Content-Type：`multipart/form-data`

#### 表单字段

- `file`

#### 返回示例

成功：

```json
{
  "error_message": "success",
  "photo": "http://127.0.0.1:3000/avatars/uuid.png"
}
```

失败：

```json
{
  "error_message": "具体错误原因"
}
```

#### 前端联调要点

- 字段名必须是 `file`。
- 文件格式只支持 `jpg/jpeg/png`。
- 上传成功后直接用返回的 `photo` 覆盖前端头像地址即可，不必再自己拼 URL。
- 如果你用了 axios/fetch，注意不要手动把 `multipart/form-data` 的 boundary 写死，让浏览器自动生成。

## 5. Bot 管理页

---

### 5.1 获取当前用户 Bot 列表

#### 适用页面

- Bot 管理页
- 匹配前 Bot 选择下拉框

#### 接口

- 方法：`GET`
- 地址：`/api/user/bot/getlist/`
- 鉴权：需要 JWT

#### 返回示例

```json
[
  {
    "id": 1,
    "userId": 100,
    "title": "稳健Bot",
    "description": "优先找空位",
    "content": "public class Bot { ... }",
    "language": "java",
    "createtime": "2026-04-15 20:00:00",
    "modifytime": "2026-04-15 20:00:00"
  }
]
```

#### 前端联调要点

- 这是数组，不是 `{data: [...]}`。
- 如果你页面要做 Bot 下拉选择，需要自行在前端补一个“手动出招”选项，通常把 `bot_id = -1` 作为手动玩家。

---

### 5.2 新增 Bot

#### 适用页面

- Bot 创建弹窗
- Bot 编辑器页

#### 接口

- 方法：`POST`
- 地址：`/api/user/bot/add/`
- 鉴权：需要 JWT

#### 请求参数

- `title`
- `description`
- `content`
- `language`

#### 可选语言值

- `java`
- `python`
- `cpp`
- `javascript`

#### 返回示例

成功：

```json
{
  "error_message": "success"
}
```

失败：

```json
{
  "error_message": "具体错误原因"
}
```

#### 前端联调要点

- `description` 传空时后端会自动补默认值。
- `language` 不传时后端默认 `java`，但前端最好显式传，避免状态不清楚。
- 创建成功后建议立即重新拉一次 Bot 列表。
- 当前后端限制每个用户最多 10 个 Bot。

---

### 5.3 更新 Bot

#### 适用页面

- Bot 编辑弹窗
- Bot 详情编辑页

#### 接口

- 方法：`POST`
- 地址：`/api/user/bot/update/`
- 鉴权：需要 JWT

#### 请求参数

- `bot_id`
- `title`
- `description`
- `content`
- `language`

#### 返回示例

成功：

```json
{
  "error_message": "success"
}
```

失败：

```json
{
  "error_message": "具体错误原因"
}
```

#### 前端联调要点

- 参数 `bot_id` 必须传。
- 更新成功后重新拉列表最稳妥。
- 如果一个 Bot 已被删除或不属于当前用户，后端会返回错误。

---

### 5.4 删除 Bot

#### 适用页面

- Bot 管理页

#### 接口

- 方法：`POST`
- 地址：`/api/user/bot/remove/`
- 鉴权：需要 JWT

#### 请求参数

- `bot_id`

#### 返回示例

成功：

```json
{
  "error_message": "success"
}
```

失败：

```json
{
  "error_message": "Bot不存在或无权限等错误"
}
```

#### 前端联调要点

- 删除成功后应从当前列表中移除，或直接重新拉一次列表。
- 如果删除的是当前匹配页里已选中的 Bot，前端需要同步重置选中状态。

## 6. 排行榜页

---

### 6.1 获取排行榜

#### 适用页面

- 排行榜页

#### 接口

- 方法：`GET`
- 地址：`/api/ranklist/getlist/`
- 鉴权：需要 JWT

#### 请求参数

- `page`

#### 请求示例

```text
/api/ranklist/getlist/?page=1
```

#### 返回示例

```json
{
  "users": [
    {
      "id": 1,
      "username": "player1",
      "password": "",
      "photo": "头像URL",
      "rating": 1700
    }
  ],
  "users_count": 123
}
```

#### 前端联调要点

- 每页固定 10 条，前端分页器可以按这个规则做。
- `users_count` 是总人数，不是总页数；总页数需要前端自己算。
- 后端返回里有 `password` 字段，但已经被清空，前端直接忽略即可。

#### 页面建议展示字段

- 排名序号
- 用户头像
- 用户名
- Rating

## 7. 对局记录页

---

### 7.1 获取战绩列表

#### 适用页面

- 战绩列表页
- 首页最近对局模块

#### 接口

- 方法：`GET`
- 地址：`/api/record/getlist/`
- 鉴权：需要 JWT

#### 请求参数

- `page`

#### 返回示例

```json
{
  "records": [
    {
      "a_photo": "A头像URL",
      "a_username": "A用户名",
      "b_photo": "B头像URL",
      "b_username": "B用户名",
      "result": "A胜",
      "record": {
        "id": 1,
        "aId": 1,
        "aSx": 11,
        "aSy": 1,
        "bId": 2,
        "bSx": 1,
        "bSy": 12,
        "aSteps": "0123",
        "bSteps": "1230",
        "map": "001011...",
        "loser": "B",
        "createtime": "2026-04-15 20:00:00"
      }
    }
  ],
  "records_count": 456
}
```

#### 前端联调要点

- 列表页直接用 `a_username`、`b_username`、`a_photo`、`b_photo`，不需要自己再查用户信息。
- 跳转回放详情页时，用 `record.id` 作为主键。
- `records_count` 是总记录数，前端自己算分页。

---

### 7.2 获取单局详情

#### 适用页面

- 对局回放页
- 对局复盘弹窗

#### 接口

- 方法：`GET`
- 地址：`/api/record/get/`
- 鉴权：需要 JWT

#### 请求参数

- `record_id`

#### 返回示例

```json
{
  "error_message": "success",
  "record": {
    "id": 1,
    "aId": 1,
    "aSx": 11,
    "aSy": 1,
    "bId": 2,
    "bSx": 1,
    "bSy": 12,
    "aSteps": "0123",
    "bSteps": "1230",
    "map": "001011...",
    "loser": "A",
    "createtime": "2026-04-15 20:00:00"
  }
}
```

#### 前端联调要点

- `map` 是压平的一维字符串，不是二维数组；回放页需要自己重建地图。
- `aSteps` 和 `bSteps` 是方向串，要按和游戏实时对局一致的规则重放。
- `loser` 只告诉谁输了，前端若要显示“谁赢了/平局”，需要自己做翻译。

## 8. 匹配大厅与对战页

这部分是前端联调最关键的地方，因为它不是单纯 HTTP，而是“主 WebSocket + 内部异步消息推送”的模式。

### 8.1 连接主对战 WebSocket

#### 连接地址

```text
ws://127.0.0.1:3000/websocket/{token}
```

#### 适用页面

- 匹配大厅
- 游戏对战页

#### 前端联调要点

- 必须先登录拿到 token。
- token 无效时服务端会直接关闭连接。
- 这条连接通常从进入大厅开始就建立，并在整个匹配/对局期间保持。

---

### 8.2 开始匹配

#### 客户端发送

```json
{
  "event": "start-matching",
  "bot_id": -1
}
```

#### 字段说明

- `event`：固定为 `start-matching`
- `bot_id`：
- `-1` 表示手动玩家
- 正整数表示选中的 Bot ID

#### 适用页面

- 匹配大厅页的“开始匹配”按钮
- 游戏房间页的“重新匹配”按钮

#### 前端联调要点

- 开始匹配前要先确认用户选中的 Bot。
- 没有选 Bot 时，前端就传 `-1`。
- 发出后前端通常会切到“匹配中”状态，并允许用户点“取消匹配”。

---

### 8.3 取消匹配

#### 客户端发送

```json
{
  "event": "stop-matching"
}
```

#### 适用页面

- 匹配中状态下的取消按钮

#### 前端联调要点

- 发完后前端即可回到未匹配状态，不需要等待额外 HTTP 回包。
- 真正是否移出队列由后端内部去调 `matchingsystem` 完成。

---

### 8.4 匹配成功并开始游戏

#### 服务端推送

```json
{
  "event": "start-matching",
  "opponent_username": "对手名",
  "opponent_photo": "对手头像",
  "game": {
    "a_id": 1,
    "b_id": 2,
    "a_sx": 11,
    "a_sy": 1,
    "b_sx": 1,
    "b_sy": 12,
    "map": [[0,0,1], [1,0,0]]
  },
  "room_id": "1_2"
}
```

#### 前端联调要点

- 事件名仍然叫 `start-matching`，但语义已经是“匹配成功，开局”。
- 前端应根据 `a_id`、`b_id` 和当前用户 ID 判断自己是 A 还是 B。
- `game.map` 这里是二维数组，可直接用来初始化实时对局地图。
- 收到 `room_id` 后，如果你有聊天功能，应立刻去连接聊天 WebSocket。
- 同时应退出“匹配中”UI，进入“对战中”UI。

---

### 8.5 手动玩家移动

#### 客户端发送

```json
{
  "event": "move",
  "direction": 0
}
```

#### 方向约定

通常是：

- `0`
- `1`
- `2`
- `3`

具体方向语义以前端地图逻辑为准，但必须和后端、Bot 代码保持一致。

#### 前端联调要点

- 只有手动玩家需要发这个消息。
- 如果这局是 Bot 在下，前端不要发送 `move`，发了也不会生效。
- 如果游戏处于暂停状态，后端会忽略移动输入。

---

### 8.6 一回合移动广播

#### 服务端推送

```json
{
  "event": "move",
  "a_direction": 0,
  "b_direction": 2
}
```

#### 前端联调要点

- 这才是前端真正推进一回合动画的依据。
- 不要在玩家本地一按键就立即最终落地，而应以服务端广播为准，避免和对手状态不一致。
- 前端需要根据这个消息同时更新 A、B 两条蛇的位置。

---

### 8.7 暂停游戏

#### 客户端发送

```json
{
  "event": "pause"
}
```

#### 服务端广播

```json
{
  "event": "game-paused",
  "paused_by": 1
}
```

#### 前端联调要点

- 收到后双方都应该进入暂停态。
- 可以在 UI 上提示“某某玩家已暂停”。

---

### 8.8 恢复游戏

#### 客户端发送

```json
{
  "event": "resume"
}
```

#### 服务端广播

```json
{
  "event": "game-resumed"
}
```

#### 前端联调要点

- 只有发起暂停的一方本人恢复才会成功。
- 前端发送后如果没收到 `game-resumed`，说明恢复条件不满足。

---

### 8.9 对局结束

#### 服务端推送

```json
{
  "event": "result",
  "loser": "A"
}
```

#### `loser` 取值

- `A`
- `B`
- `all`

#### 前端联调要点

- 根据当前用户是 A 还是 B，翻译成“胜利 / 失败 / 平局”。
- 后端在发这条消息前已经完成积分更新和战绩写库。
- 对战结束页如果要展示最新积分，建议再刷新一次用户信息或排行榜相关数据。

## 9. 对局聊天功能

### 9.1 连接聊天 WebSocket

#### 连接地址

```text
ws://127.0.0.1:3003/chat/{token}/{roomId}
```

#### `roomId` 来源

来自主对战 WebSocket 开局消息中的：

```json
"room_id": "1_2"
```

#### 前端联调要点

- 聊天连接通常在收到开局消息后再创建。
- 如果用户退出对局页，可以关闭这条连接。

---

### 9.2 发送聊天消息

#### 客户端发送

```json
{
  "content": "你好",
  "username": "playerA"
}
```

#### 前端联调要点

- `content` 为空或全是空格时，后端会忽略。
- `username` 是前端传给聊天服务的显示名；建议直接使用当前登录用户名。

---

### 9.3 接收聊天广播

#### 服务端广播

```json
{
  "userId": 1,
  "username": "playerA",
  "content": "你好",
  "timestamp": 1713180000000
}
```

#### 前端联调要点

- `userId` 可以用来区分“自己发的消息”与“对手发的消息”。
- `timestamp` 是毫秒时间戳，前端自行格式化显示。
- 同房间内所有连接都会收到，包括发送者自己。

## 11. 联调中最容易出错的点

### 11.1 参数名不统一

这个项目有几组字段名非常容易写错：

- 注册：`confirmedPassword`
- 改用户名：`new_username`
- 改密码：`confirmed_password`
- 删除/更新 Bot：`bot_id`
- 单局详情：`record_id`

前端如果用了统一表单模型，需要特别留意字段映射。

### 11.2 默认把 POST 当 JSON 发

很多前端封装默认用 JSON body，但这个项目后端控制器基本是 `@RequestParam`。如果你直接发 JSON，后端很可能取不到字段。

### 11.3 混淆两个 WebSocket

- `3000/websocket/{token}`：主对战消息
- `3003/chat/{token}/{roomId}`：聊天消息

它们职责完全不同，不要混在一起。

### 11.4 对局开始事件命名容易误判

服务端推送的开局事件名是：

```json
"event": "start-matching"
```

它字面像“开始匹配”，但实际表示“匹配成功，开始对局”。前端事件处理一定要按当前实现适配。

### 11.5 战绩列表和单局详情不是一个结构

- 列表接口返回的是包装过的记录项
- 单局详情返回的是单个 `record`

不要指望两个接口字段完全一样。

## 12. 推荐的前端联调顺序

如果你准备从零开始联调，建议按这个顺序：

1. 登录 / 注册
2. 当前用户信息
3. Bot 管理 CRUD
4. 排行榜
5. 战绩列表 + 单局详情
6. 主 WebSocket 连接
7. 匹配开始 / 取消
8. 对战移动与结算
9. 对局聊天
10. 个人资料修改与头像上传

这样做的原因是：

- 先把登录态跑通，后面所有鉴权接口都依赖它
- 先把 Bot 管理跑通，匹配页才能选择 Bot
- 最后再联调 WebSocket，对问题定位更容易

## 13. 总结

从前端联调角度看，这个项目最核心的不是 HTTP CRUD，而是三条并行能力：

- 账号与 Bot 管理的标准 HTTP 接口
- 主对战 WebSocket 的实时状态同步
- 聊天 WebSocket 的房间消息广播

前端只要把下面这几个关键点处理对，联调会顺很多：

- token 放在 HTTP header，WebSocket 放在路径里
- POST 参数按表单字段名传，不要想当然改名
- 对战逻辑以服务端 WebSocket 广播为准，不要只看本地输入
- 聊天房间号使用主对战 WebSocket 返回的 `room_id`
- 开局事件虽然叫 `start-matching`，但要按“进入游戏”处理

这份清单适合作为前端页面接接口时的直接参考版本。
